## 1. なぜこのテーマが重要なのか（動機）

### ブロックチェーンソフトウェアのバグは「極めて」高コスト

ブロックチェーンの世界では、ソフトウェアのバグ（プログラムの欠陥）が発見・悪用されると、**数百億円規模の損失**が発生することがあります。

ブロックチェーンシステムは、下から順に次のような層（レイヤー）で構成されています。

```
Web3（ユーザー向けアプリ）
   ↑
DeFiアプリ（分散型金融）
   ↑
ゼロ知識回路（暗号技術の核心部分）
   ↑
ブロックチェーンプロトコル（基盤）
```

**どの層にバグがあっても、悪用されれば壊滅的な被害が生じます。**

### 実際の被害事例

| 被害事例 | バグの種類 | 層 |
|---------|-----------|-----|
| Beanstalk（DeFiプロトコル）$1.82億ドル被害 | フラッシュローン脆弱性 | スマートコントラクト |
| Solana ネットワーク停止 | DoS脆弱性（コンセンサスプロトコル） | ブロックチェーンプロトコル |
| Zcash（匿名通貨）の偽造可能バグ | 算術回路のバグ（zkSNARK実装） | ゼロ知識回路 |

特に **Zcash のケース**は深刻で、2018年3月に発見されたバグを放置すると、攻撃者が**偽のZcashコインを大量に生成**できる状態でした。これはまさにZK回路（ゼロ知識証明を実装する回路）に存在したバグです。

## 2. フォーマルメソッドとは何か

### 定義

**フォーマルメソッド（Formal Methods、形式手法）**とは、ソフトウェアについて：
- **バグを発見する**
- **正しく動くことを数学的に証明する**

ための、**数学的に厳密な技術・手法の総称**です。

「テストをたくさん実行してバグがないか確認する」という従来の方法と異なり、フォーマルメソッドは**すべての可能な動作を数学的に網羅**しようとします。

### フォーマルメソッドの技術スペクトラム

フォーマルメソッドには複数の技術があり、「保証の強さ」と「必要な人間の手間」のトレードオフがあります。

```
弱い保証 ←──────────────────────────────→ 強い保証
少ない手間 ←───────────────────────────→ 多い手間

[ファジング] [コンコリック実行] [抽象解釈] [形式検証]
```

#### ① ファジング（Fuzzing）＝動的手法
- ランダムな入力値を大量にプログラムに与えて、クラッシュや異常を検出する
- 手軽だが、すべてのバグを見つけられるわけではない

#### ② コンコリック実行（Concolic Execution）＝動的手法
- 「具体的な実行」と「シンボリック（記号的）な実行」を組み合わせた手法
- より体系的にプログラムのパスを探索する

#### ③ 抽象解釈（Abstract Interpretation）＝静的手法
- プログラムのソースコードを実行せずに解析する
- すべての可能な実行パスを「抽象的に」シミュレートする

#### ④ 形式検証（Formal Verification）＝静的手法
- プログラムが仕様を満たすことを**数学的な証明として確認**する
- 最も強い保証が得られるが、最も多くの人間の作業が必要

### 動的手法 vs 静的手法

| 分類 | 方法 | 特徴 |
|------|------|------|
| **動的（Dynamic）** | ファジング、コンコリック実行 | プログラムを実際に動かして観察する |
| **静的（Static）** | 抽象解釈、形式検証 | ソースコードを解析してすべての実行を推論する |

## 3. ZK回路とは何か・なぜバグが危険なのか

### ZK回路のワークフロー

ゼロ知識証明（Zero-Knowledge Proof, ZKP）を実際に使うには、**ZK回路**という特殊なプログラムを書く必要があります。開発の流れは次の通りです。

```
①ソースコード（Circom / Halo2で記述）
        ↓ コンパイル
②2種類のもの（同時に生成）：
   - 証明者（Witness Generator）P：「入力 x を受け取り、出力 y を計算する」
   - 制約（Constraints）C：「(x, y) の組が正しいか検証する多項式方程式」
        ↓
③SNARK（簡潔な非対話型証明）
   - Prover（証明者）：証明を生成
   - Verifier（検証者）：証明を確認
```

**重要な前提**：証明者（P）と制約（C）は**同等（等価）でなければなりません**。

### 等価性（Equivalence）とは何か

等価性は次の数学的条件で定義されます。

$$\forall x, y. \quad P(x) = y \iff C(x, y) = \text{true}$$

「すべての入力 $x$ と出力 $y$ について、プログラム $P$ が $x$ を入力して $y$ を出力するとき、かつそのときに限り、制約 $C(x, y)$ が真である」

これは2つの要件に分解できます。

**要件①（左→右方向）**：

$$\text{すべての } (x, y) \text{ で } P(x) = y \text{ ならば } C(x, y) = \text{true}$$

*Pの正しい入出力ペアはすべてCを満たす*

**要件②（右→左方向）**：

$$\text{すべての } (x, y) \text{ で } C(x, y) = \text{true} \text{ ならば } P(x) = y$$

*Cを満たすすべての(x,y)はPの正しい入出力ペアである*

### 等価性違反のパターン

#### 過制約バグ（Overconstrained Bug）

$$\exists x, y \text{ で } P(x) = y \text{ だが } C(x, y) = \text{false}$$

- 制約が**厳しすぎる**
- 正しい入出力ペアなのに、制約が拒否してしまう
- 「合法な証明が通らない」問題

#### 過小制約バグ（Underconstrained Bug）— **最も危険！**

$$\exists x, y \text{ で } C(x, y) = \text{true} \text{ だが } P(x) \neq y$$

- 制約が**緩すぎる**
- プログラムが出力しないはずの偽の $(x, y)$ ペアを、制約が「正しい」と認めてしまう
- 「不正な証明が通ってしまう」問題

### なぜ過小制約バグが致命的なのか

ZKシステムでは、**検証者（Verifier）は制約 $C$ を使って証明を確認します**。

過小制約バグがあると：
- 攻撃者が「Pが計算したことのない偽の出力 $y$」を $C$ に通すことができる
- Verifier は $C$ が満たされているので「正しい」と判断してしまう
- 結果として：**二重支払い（Double Spend）** や **トークンの大量詐取** が可能になる

実例として、Tornado Cash（Ethereum上のプライバシープロトコル）がこのバグで攻撃されました。また、Aztec 2.0でも `tree_index` 変数の範囲制約が欠落していたバグが発見されています。

## 4. ZKバグの分類

ZK回路のバグは大きく3種類に分類されます。

```
ZK回路のバグ
├── ① 未制約シグナル（Unconstrained Signals）
│   ├── 未制約出力（Unconstrained Output）
│   └── 未制約パブリックシグナル（Unconstrained Public Signal）
├── ② 安全でないコンポーネント使用（Unsafe Component Usage）
│   ├── 過小制約サブ回路出力（Under-Constrained Sub-Circuit Output）
│   └── 過小制約サブ回路入力（Under-Constrained Sub-Circuit Input）
└── ③ 制約/計算の不一致（Constraint/Computation Discrepancy）
    └── ゼロ逆数なし（No Zero Inverse）
```

---

### ① 未制約シグナル（Unconstrained Signals）

**定義**: あるシグナル（変数）に対して、制約が常に真になってしまう（何でも受け入れてしまう）状態。

#### 実例：未制約出力（Circomコード）

```circom
template Num2Bits(n) {
    signal input in;
    signal output out[n];
    var lc1 = 0;
    var e2 = 1;
    for (var i = 0; i < n-1; i++) {  // ← バグ: n-1 で止まっている（n であるべき）
        out[i] <-- (in >> i) & 1;
        out[i] * (out[i] - 1) === 0;
        lc1 += out[i] * e2;
        e2 = e2 + e2;
    }
    lc1 === in;
}
```

**何をするコードか？**

`Num2Bits(n)` は「入力 `in` を n ビットの2進数に分解する」テンプレートです。例えば `in = 5`、`n = 3` なら `out = [1, 0, 1]`（= 101₂）を出力します。

**$n = 3$ のとき、生成される制約**：

$$\text{input: } in$$

$$\text{output: } out_0, out_1, out_2$$

$$out_0 \cdot (out_0 - 1) = 0$$

$$out_1 \cdot (out_1 - 1) = 0$$

$$out_0 + 2 \cdot out_1 = in$$

**バグの説明**：

ループが `i < n-1`（つまり `i < 2`）で止まるため、最後のビット $out_2$ に対する制約が追加されません。$out_2$ は**制約がないため、任意の値を取ることができます**。

攻撃者は $out_2$ に好きな値を設定した偽の証明を生成し、Verifier を騙せます。

---

### ② 安全でないコンポーネント使用（Unsafe Component Usage）

サブ回路（部品として使い回す回路テンプレート）は、**入力・出力に適切な制約が置かれることを前提に設計**されています。これを守らずに使用すると脆弱性が生まれます。

#### 実例：過小制約サブ回路出力

```circom
template withdraw(n) {
    assert(n <= 252);
    signal input bal;   // 残高（balance）
    signal input amt;   // 引き出し額（amount）
    signal output out;

    component n2b1 = Num2Bits(n); // bal < 2^n を確認
    n2b1.in <== bal;

    component n2b2 = Num2Bits(n); // amt < 2^n を確認
    n2b2.in <== amt;

    component lt = LessThan(n); // amt < bal を確認するサブ回路
    lt.in[0] <== bal;
    lt.in[1] <== amt;

    // ⚠️ バグ: lt.out === 0 という制約が抜けている！

    out <== bal - amt;
}
```

**何をするコードか？**

これは「残高 `bal` から `amt` を引き出す」処理です。正しく動作させるには `amt < bal`（引き出し額が残高を超えていないこと）を検証しなければなりません。

`LessThan(n)` サブ回路は `lt.out` に比較結果（1なら真、0なら偽）を返しますが、**その出力を制約に使っていません**。

**正しくは**：

```circom
lt.out === 1;  // amt < bal が成立していることを強制する
```

**バグの影響**：

この制約がないため、`amt > bal`（残高より多い額）を引き出す不正な証明も Verifier が通してしまいます。攻撃者は**持っている以上の資金を引き出せます**。

---

### ③ 制約/計算の不一致（Constraint/Computation Discrepancy）

制約（`===`）はフィールド上の多項式方程式なので、すべての計算を直接表現できるわけではありません。計算の意味が制約に正確に反映されていない場合に発生するバグです。

#### 実例：ゼロ逆数なし（No Zero Inverse）

```circom
template MulInverse() {
    signal input a;
    signal input b;
    signal output out;

    out <-- a / b;        // 計算: out = a ÷ b
    out * b === a;        // 制約: out × b = a
}
```

**問題点**：

数学的に、$b = 0$ のとき $a \div b$ は**定義されません**（ゼロ除算）。しかし制約 $out \times b = a$ は、$a = 0, b = 0$ のとき：

$$out \times 0 = 0 \quad \Rightarrow \quad 0 = 0$$

これはどんな `out` の値でも成立してしまいます。つまり **$a = 0, b = 0$ のとき、任意の `out` を持つ偽の証明が Verifier を通過します**。

## 5. 静的解析ツール：Vanguard

### Vanguard の仕組み

Vanguard は Veridise 社が開発した静的解析ツールで、次の2段階で動作します。

```
ソースコード（Circom / Halo2）
        ↓
[ステップ1] CDG（回路依存グラフ）を構築
        ↓
[ステップ2] CDG を解析して脆弱性を検出
        ↓
脆弱性レポート
```

### CDG（Circuit Dependence Graph）とは

CDG は「計算と制約の関係」を表すグラフです。

- **計算の依存関係**（`<--`）：実線矢印
  - `o <-- i` → i から o に実線矢印
- **制約の依存関係**（`===`）：点線矢印
  - `o === i` → i から o に点線矢印

計算がある（実線）のに対応する制約がない（点線がない）シグナルは、**未制約（Underconstrained）の候補**として検出されます。

### 静的解析の基本原理：なぜ完全には判定できないのか

プログラムのすべての実行パスを正確に求めることは**数学的に不可能**（停止問題・ライスの定理）であることが知られています。そのため静的解析は「実際の状態の集合」を**過近似**します。

```
┌──────────────────────────────┐
│       全可能状態              │
│   ┌──────────────┐          │
│   │   静的解析が  │  ← 過近似（緑の楕円）
│   │   計算した状態 │          │
│   │  ┌────────┐  │          │
│   │  │実際の状態│  │          │
│   │  │（青い形）│  │          │
│   │  └────────┘  │      ⬤ Bad states（赤）
│   └──────────────┘          │
└──────────────────────────────┘
```

- **過近似（Over-approximation）**: 実際の状態を包含するより大きな集合で近似
  - 緑の楕円が赤（Bad states）と重ならない → **バグなし**を証明できる
  - 緑の楕円が赤と重なる → **誤検出（False Positive）の可能性**あり
- **過少近似（Under-approximation）**: 実際より小さな集合で近似
  - **見逃し（False Negative）**が生じる → バグがあっても検出できない

### Vanguard の評価結果

GitHub上の17の公開Circomプロジェクトから258の回路を分析した結果：

| バグの種類 | 検出数（概数） |
|-----------|--------------|
| 未制約シグナル | 約25件 |
| 安全でないコンポーネント使用 | 約5件 |
| **制約/計算の不一致** | **約55件（最多！）** |

**合計32の未知の脆弱性を発見しました。**

→ 開発者は「計算の意味を制約に正確に反映する」ことが最も難しいと分かります。

## 6. 形式検証ツール：Picus

### 背景：既存手法の限界

ZK回路の過小制約バグを検出するための既存手法には2つあります。

#### 手法A：制約の静的解析（SA: Static Analysis）

ルールベースで制約を素早く解析する。

例：制約

$$z = 3x + 4, \quad y = z + 2x$$

$y$ は $x, z$ の線形結合なので、$y$ は一意に決まる → **即座に「過小制約でない」と判定可能**

*長所*：速い  
*短所*：判定できないケースが多い（不精密）

#### 手法B：SMTソルバー

過小制約の問題を**充足可能性（SAT）問題**として定式化します。

$$\exists y_1, y_2. \quad P[y_1/y] \land P[y_2/y] \land y_1 \neq y_2$$

「同じ入力 $x$ に対して、制約を満たす出力が2つ（$y_1 \neq y_2$）存在するか？」

もし SAT（解が存在する）なら → 過小制約バグあり  
UNSAT（解が存在しない）なら → 過小制約バグなし

*長所*：正確  
*短所*：大規模な回路では非常に遅い

### Picus のアプローチ：2つの手法を組み合わせる

Picus は静的解析（SA）と SMT ソルバーを**ループ的に連携**させることで、速さと正確さを両立します。

```
[入力] 多項式フィールド方程式 P, 一意性が証明済みのシグナル集合 K = {}
         ↓
┌──────────────────────────────────────────┐
│ ループ開始                                 │
│                                          │
│  [SA フェーズ]                            │
│   P, K を入力 → K' を出力                │
│   (K ⊆ K': より多くのシグナルが一意と証明) │
│         ↓                                │
│   OutputSignals ⊆ K' なら → ✅ 安全！    │
│   そうでなければ → SMT フェーズへ          │
│                                          │
│  [SMT フェーズ]                           │
│   P, K を入力 → K'', K_uncons を出力      │
│   K'' computed assuming K (Kが大きいほど速い) │
│         ↓                                │
│   OutputSignals ⊆ K'' なら → ✅ 安全！   │
│   OutputSignals ∩ K_uncons ≠ ∅ なら → ❌ バグ！ │
│   K = K'' なら → ?? 不明（タイムアウト）  │
│   そうでなければ K'' を SA フェーズに戻す  │
└──────────────────────────────────────────┘
```

**なぜこれが賢いのか？**

- 静的解析で「一意と証明できたシグナルの集合 $K$」を SMT に渡す
- SMT は「$K$ のシグナルは固定値と仮定して」クエリを解く → $K$ が大きいほど**SMT クエリが速くなる**
- SMT の結果（$K''$）でさらに一意なシグナルが増えれば、それを SA に戻してさらに解析を進める

### Picus の出力例

```
$ ./picus-solve.sh ./benchmarks/motivating/adder.r1cs

# number of constraints: 9
# parsing alternative r1cs...
# configuring precondition...
safe.
```

`safe.` という出力が出れば、**この回路に過小制約シグナルがないことが数学的に保証されます。**

### Picus の評価結果

| ベンチマークセット | 回路数 | 平均制約数 | 平均出力シグナル数 |
|-----------------|-------|----------|----------------|
| circomlib-utils | 59 | 352 | 10 |
| circomlib-core | 104 | 6,690 | 32 |
| **合計** | **163** | **4,396** | **24** |

Picus（青のバー）は競合手法（SA単独、SMT単独）を上回る解決率を達成しています。

- circomlib-utils では **83%** の回路を解決
- circomlib-core では **63%** の回路を解決

> **DeIn への示唆**  
> Ethereum Sepolia 上で動作する DeIn プロトコルは、パラメトリック地震保険の払い出しロジックを Solidity スマートコントラクトで実装していますが、将来的に ZK 証明（例：プライベートな保険金請求の検証）を組み込む際には、上記のような ZK 回路のバグ——特に**過小制約バグ**——が致命的なセキュリティリスクになります。Vanguard や Picus のような形式検証ツールの導入は、DeIn の安全性保証において重要な検討事項となります。
