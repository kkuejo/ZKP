## 1. 全体像：ZKPをどうやって使うのか

### ZKPとは何か（おさらい）

**ゼロ知識証明（Zero-Knowledge Proof, ZKP）** とは、ある秘密の情報を**相手に一切開示せずに**、「その情報を知っている」という事実だけを証明できる仕組みです。

#### 具体例で理解する

- **パスワードの証明**: パスワードそのものを送らずに「このパスワードを知っている」と証明できる
- **年齢確認**: 生年月日を見せずに「18歳以上である」と証明できる
- **残高確認**: 口座番号や残高を開示せずに「残高が100万円以上ある」と証明できる

### ZKPを使う際のパイプライン

```
プログラマ → コンパイラ → セットアップ → 証明生成
  アイデア  →  プログラム → R1CS → パラメータ → ZKP（証明）
```

**この講義（Lecture 3）は、アイデアをR1CSに変換する部分を扱います。**

## 2. ZKPのプログラム可能性

### 2.1 ZKPの基本構造

ZKPには3つの登場人物がいます：

| 役割 | 知っていること | やること |
|------|--------------|---------|
| **証明者（Prover）** | $x$（公開情報）、$w$（秘密）、$\phi$（命題） | 証明 $\pi$ を生成 |
| **検証者（Verifier）** | $x$（公開情報）、$\phi$（命題） | 証明 $\pi$ を検証 |

数学的に書くと：

$$\phi(x, w) = \text{true}$$

という命題が成り立つことを、$w$（**witness：秘密の証人**）を開示せずに証明します。

### 2.2 命題 $\phi$ は何になるのか

**理論上の例:**
- $w$ は整数 $x$ の素因数分解である
- $w$ は公開鍵 $x$ に対応する秘密鍵である
- $w$ は口座 $x$ の正当な認証情報である

**実際の実装:**
- $\phi$ は「$x$と$w$を入力とする**算術回路（Arithmetic Circuit）**」として表現される

## 3. 算術回路（Arithmetic Circuits）

### 3.1 素体（Prime Field）とは

算術回路の計算は、通常の整数ではなく**「素体（Prime Field）」**上で行われます。

**素体** $\mathbb{Z}_p$ とは：
- $p$ は巨大な素数（約255ビット、つまり $2^{254}$ 程度の大きさ）
- 整数を $p$ で割った余りの世界（**mod $p$** の演算）
- 使える演算：加算 $+$、乗算 $\times$、等号 $=$（すべて mod $p$）

**例：$\mathbb{Z}_5$（$p=5$の場合）**

$$4 + 5 \equiv 9 \equiv 4 \pmod{5}$$
$$4 \times 4 \equiv 16 \equiv 1 \pmod{5}$$

通常の計算と違い、数が「循環」するイメージです。なぜこのような体を使うのかというと、ZKPの暗号数学的な安全性のためです。

### 3.2 算術回路の2つの表現

#### 表現①：連立方程式のシステム

$$x_1 \times x_1 \times x_1 = y$$
$$x_2 \times x_2 = z$$

のような方程式の集まりとして表現できます。

#### 表現②：回路図（グラフ）

- **有向非巡回グラフ（DAG）**: 矢印が循環しない有向グラフ
- **ノード**: 入力値、ゲート（演算）、定数
- **エッジ（辺）**: ワイヤー（信号の流れ）

電子回路の設計図に似ていますが、「デジタル回路（0と1）」ではなく「算術回路（素体の元素）」を扱います。

## 4. R1CS：共通フォーマット

### 4.1 R1CSとは

**R1CS（Rank-1 Constraint System）** は、ZKPで使われる算術回路の標準的なフォーマットです。

**定義：**

- **公開入力** $x$：フィールド元 $x_1, x_2, \ldots, x_\ell$
- **秘密情報（witness）** $w$：$x_{\ell+1}, \ldots, x_n$
- **制約**：以下の形の $m$ 個の等式

$$A \times B = C$$

ただし $A$、$B$、$C$ はすべての変数の**アフィン結合**（線形結合 + 定数）

### 4.2 具体的な制約の例

$$x_i \times x_j - x_i - 1 = x_1$$
$$x_i \times x_i = x_i$$
$$x_i \times x_i \times x_i = x_1 \quad \text{（これは直接はR1CS不可）}$$

**重要な制約**: R1CSでは各制約が必ず「一方が線形 × 他方が線形 = 右辺」の形でなければなりません。

### 4.3 行列表現

より数学的に書くと：

- $\mathbf{x}$：$\ell$ 個のフィールド元のベクトル（公開入力）
- $\mathbf{w}$：$n - \ell - 1$ 個のフィールド元のベクトル（秘密）
- $A, B, C \in \mathbb{Z}_p^{m \times n}$：行列

全変数ベクトルを $\mathbf{z} = (1 \| \mathbf{x} \| \mathbf{w}) \in \mathbb{Z}_p^n$ とすると、R1CSの充足条件は：

$$A\mathbf{z} \circ B\mathbf{z} = C\mathbf{z}$$

ここで $\circ$ はアダマール積（成分ごとの乗算）です。

### 4.4 算術回路をR1CSに変換する例

**回路**: $x_1 \cdot x_2 + x_1$ を計算

**ステップ1**: 中間変数（補助変数）を導入

$$v_1 = x_1 \times x_2$$
$$\text{出力} = v_1 + x_1$$

**ステップ2**: 各乗算をR1CS制約として記述

$$x_1 \times x_2 = v_1 \quad \text{（制約1：乗算）}$$
$$\text{出力} = v_1 + x_1 \quad \text{（制約2：これは加算なので制約不要！）}$$

**ポイント**: 加算はR1CSで「タダ」（制約を消費しない）ですが、乗算は毎回1つの制約を消費します。回路の複雑さは「乗算ゲートの数」で決まります。

## 5. HDL

### 5.1 プログラミング言語（PL）とハードウェア記述言語（HDL）の違い

| 比較項目 | プログラミング言語（PL） | ハードウェア記述言語（HDL） |
|---------|----------------------|--------------------------|
| 基本単位 | 変数（Variable） | ワイヤー（Wire） |
| 処理単位 | 演算（Operation） | ゲート（Gate） |
| 構造単位 | プログラム・関数 | 回路・サブ回路 |
| 主な動作 | 変数の変更・関数呼び出し | ワイヤーの接続・サブ回路の生成 |

### 5.2 なぜHDLが必要か

ZKPプログラミングは「回路を設計する」作業です。デジタル回路設計でVerilogやVHDLが使われるのと同様に、算術回路（R1CS）の設計には専用のHDLが適しています。

### 5.3 ZKPのための開発レイヤー構造

```
高レベルの仕様
      ↓
  プログラム（高水準言語）
      ↓ ライブラリ / コンパイラ / DSL を使う
    R1CS
      ↓
   アセンブリ（ZKP証明システム）
```

## 6. Circom：R1CS のための HDL

Circomは**ZKP界のVerilog**とも言える、R1CS専用のHDLです。

### 6.1 基本言語要素

| 要素 | 意味 |
|------|------|
| `template` | サブ回路の定義（関数のようなもの） |
| `signal` | ワイヤー（R1CSの変数） |
| `input` / `output` | 入力・出力信号 |
| `<--` | 信号の**値を設定**する（制約なし） |
| `===` | R1CS**制約を作成**する |
| `<==` | 値設定 + 制約作成を同時に行う |

### 6.2 基本的な例：乗算回路

```circom
template Multiply() {
    signal input x;
    signal input y;
    signal output z;
    
    z <-- x * y;    // z の値を x*y として設定
    z === x * y;    // 制約：z = x*y であることを強制
    
    // ↑ 上2行をまとめると：z <== x * y;
}

component main {public [x]} = Multiply();
```

**重要な制限**: `===` で作れる制約は**Rank-1**（ランク1）でなければなりません：
- 左辺：**線形**（1次式）
- 右辺：**2次式**まで

```circom
z === x * y;      // OK：左辺線形、右辺2次
z === x * x * y;  // ERROR：右辺が3次（Rank-1ではない）
```

### 6.3 メタプログラミング（テンプレート引数とループ）

Circomは2つのレイヤーを持ちます：

1. **ベース言語**：回路の信号と制約を記述
2. **メタプログラミング言語**：コンパイル時に評価される（ループ・条件分岐など）

**例：繰り返し二乗法（Repeated Squaring）**

```circom
template RepeatedSquaring(n) {
    signal input x;
    signal output y;
    signal xs[n+1];   // 信号の配列
    
    xs[0] <== x;
    for (var i = 0; i < n; i++) {   // コンパイル時に展開
        xs[i+1] <== xs[i] * xs[i]; // x^(2^i) を計算
    }
    y <== xs[n];      // 最終結果：x^(2^n)
}

component main {public [x]} = RepeatedSquaring(1000);
```

`n=1000` とすると、$x^{2^{1000}}$ を計算する回路が生成されます。ループはコンパイル時に1000個の制約に展開されます。

**変数（`var`）と信号（`signal`）の違い**:

| 種類 | 変更可能 | タイミング | R1CS変数になるか |
|------|---------|----------|----------------|
| `var` | ○ | コンパイル時 | ✗ |
| `signal` | ✗ | 実行時 | ○ |

### 6.4 Witness計算とサブ回路

R1CSで直接表現できない計算（例：除算）は、**witness計算**として `<--` で記述し、その正しさを `===` で確認します。

```circom
template NonZero() {
    signal input in;
    signal inverse;
    
    inverse <-- 1 / in;     // 除算（R1CSで直接表現不可）
    1 === in * inverse;      // 検証：in * (1/in) = 1（R1CS制約）
}

template Main() {
    signal input a;
    signal input b;
    
    component nz = NonZero();  // サブ回路を使用
    nz.in <== a;               // ドット記法でアクセス
    0 === a * b;               // a*b = 0 の制約
}
```

**考え方**: 「答えを直接計算するのが難しくても、答えが正しいことを検証するのは簡単」という非対称性を活用します。

### 6.5 チュートリアル例：数独（Sudoku）

数独をZKPで証明するとは：
- **公開情報** $x$：数独のパズル（一部が埋まったグリッド）
- **秘密情報** $w$：数独の解答（全マスが埋まったグリッド）
- **命題 $\phi$**：「この解答はパズルの正しい答えである」

検証すべき条件：
1. 各マスが1〜9の値を持つ
2. 各行に重複がない
3. 各列に重複がない（講義では省略）
4. 各3×3ブロックに重複がない（講義では省略）

この証明により、解答を見せずに「解答を知っている」ことを証明できます。

## 7. R1CSライブラリ（Arkworks）

### 7.1 ライブラリ方式の概要

Circomが「専用言語」であるのに対し、**ライブラリ方式**はRustやC++などの既存言語に追加するアプローチです。

**コアとなる型**：制約システム（Constraint System）

```
制約システム
├── 変数一覧（A列、B列、C列）
└── 制約一覧（A×B = C）
```

### 7.2 基本操作

```rust
// 変数の作成
cs.add_var(p, v) → id
// p: 変数の公開/秘密の区別
// v: 変数に代入する値
// id: 変数のハンドル（参照）

// 線形結合（Linear Combination）の作成
cs.zero()           // 空の線形結合
lc.add(c, id) → lc' // lc' = lc + c * id

// 制約の追加
cs.constrain(lcA, lcB, lcC)
// lcA × lcB = lcC という制約を追加
```

### 7.3 低レベルAPI：Boolean AND の例

```rust
fn and(cs: ConstraintSystem, a: Var, b: Var) -> Var {
    // 結果変数を作成（値はa & b）
    let result = cs.new_witness_var(|| a.value() & b.value());
    
    // 制約を追加：a × b = result
    self.cs.enforce_constraint(
        lc!() + a,       // 線形結合A = a
        lc!() + b,       // 線形結合B = b
        lc!() + result,  // 線形結合C = result
    );
    
    result
}
```

これは**非常に低レベルで使いにくい**実装です。

### 7.4 高レベルAPI：言語抽象化の活用

Rustの構造体、演算子オーバーロード、メソッドなどを活用して、より使いやすいAPIを提供：

```rust
// Boolean型の定義
struct Boolean { var: Var }

// AND演算子のオーバーロード
impl BitAnd for Boolean {
    fn bitand(self: Boolean, other: Boolean) -> Boolean {
        // 内部でR1CS制約を自動生成
        Boolean { var: result }
    }
}
```

**実際の使い方：**

```rust
let a = Boolean::new_witness(|| true);   // 秘密の入力
let b = Boolean::new_witness(|| false);  // 秘密の入力

// 通常のコードと同じような書き方でR1CS制約が自動生成される！
(a & b).enforce_equal(Boolean::FALSE);
```

### 7.5 Witness計算の柔軟性

ライブラリ方式では、**クロージャ（ラムダ式）** を使って任意の計算でwitnessを生成できます：

```rust
// クロージャは証明生成時にのみ実行される
let a = Boolean::new_witness(|| (4 == 5) & (x < y));
```

これにより、除算、比較、条件分岐など、R1CSで直接表現できない計算も使えます。

### 7.6 主なR1CSライブラリ

| ライブラリ | 言語 | 特徴 |
|-----------|------|------|
| **Arkworks** (r1cs-std) | Rust | 最新・型安全・豊富なガジェット |
| **libsnark** (gadgetlib) | C++ | 初期の主要実装 |
| **Bellman** | Rust | Zcashで使用 |
| **Snarky** | OCaml | 関数型言語ベース |
| **Gnark** | Go | Go言語向け |

## 8. コンパイラ方式（ZoKrates）

### 8.1 コンパイラ方式の発想

HDLやライブラリでは「ワイヤーと制約を明示的に記述」する必要があります。

**コンパイラ方式**では、普通のプログラムを書くと、自動的にR1CSに変換してくれます。

```
通常のプログラム  → コンパイラ → R1CS
（変数・ループ）               （ワイヤー・制約）
```

### 8.2 ZoKratesの構文

```zokrates
type F = field;

def main(public F x, private F[2] ys) {
    field y0 = ys[0];
    field y1 = ys[1];
    assert(x == y0 * y1);  // 制約を生成
}
```

**キーワード:**
- `public`：公開入力
- `private`：秘密入力（witness）
- `assert`：R1CS制約を生成
- `field`：素体の元素型

### 8.3 ZoKratesの言語機能

```zokrates
// 整数ジェネリクスとループ
def repeated_squaring<N>(field x) -> field {
    field[N+1] mut xs;  // 可変配列
    xs[0] = x;
    
    for u32 i in 0..N {  // 固定長ループ
        xs[i + 1] = xs[i] * xs[i];
    }
    
    return xs[N];
}

def main(public field x) -> field {
    return repeated_squaring::<1000>(x);
}
```

**ZoKratesでサポートされる機能:**
- 整数ジェネリクス（`<N>`）
- 配列（固定長）
- 可変変数（`mut`）
- 固定長ループ（`for ... in 0..N`）
- if式
- 構造体（カスタム型）

### 8.4 ZoKratesの制限：Witness計算

ZoKratesの**最大の制限**は、witness計算の柔軟性がないことです：

```zokrates
// 逆数を求める場合、bをwitnessとして外部から提供しなければならない
def main(private field a, public field b) {
    assert(a * b == 1)
    // b = 1/a を自分で計算することができない
}
```

CircomやArkworksと異なり、witnessの計算ロジックを記述する手段がないため、すべての秘密入力を外部から与える必要があります。

## 9. ZKPツールチェーン全体像

### 9.1 ツールチェーンの分類

```
                  独立言語か？
                 /          \
              No              Yes
              |               |
         回路ライブラリ    言語の種類
         (arkworks)       /        \
                       HDL         PL（プログラム言語）
                    (circom)         (Noir等)
```

### 9.2 主なツールチェーンの比較

#### ZoKrates（PL + コンパイラ方式）

**長所:**
- 最も学びやすい
- エレガントな構文（普通のプログラムに近い）

**短所:**
- Witness計算の柔軟性が低い
- 制約の制御が難しい

#### Arkworks（ライブラリ方式）

**長所:**
- 制約が明確
- Rustの表現力をそのまま活用できる

**短所:**
- Rustの知識が必要
- 最適化の自動化が少ない

#### Circom（HDL方式）

**長所:**
- 制約が明確に見える
- エレガントな構文

**短所:**
- 学習が難しい
- 抽象化が限定的

### 9.3 ツールチェーンの歴史

```
2015年                      2020年以降
|                              |
↓                              ↓
Gadgetlib → Bellman → Snarky → Arkworks → PLONKish
           Circom → Leo
           ZoKrates → Noir
                    Cairo
```

### 9.4 共有コンパイラ基盤：CirC

多くのZKP言語は共通の処理が必要です（ブール演算、固定幅整数、RAM検査、制御フロー等）。

**CirC**（github.com/circify/circ）は、これらの共通処理を提供するライブラリで、複数のZKP言語コンパイラから利用できます。

| ソース言語 | ターゲット形式 | 共通処理 |
|-----------|--------------|---------|
| Circom | R1CS | ブール演算 |
| ZoKrates | PLONK | 固定幅整数 |
| Noir | AIR | RAM検査 |
| Cairo | | 変数・制御フロー最適化 |

### 9.5 ターゲット（証明システム）の種類

| 形式 | 主な使用例 | 特徴 |
|------|-----------|------|
| **R1CS** | Groth16, Marlin | 最も一般的 |
| **PLONK** | PlonKish系 | より柔軟な制約 |
| **AIR** | STARKs | 再帰的証明に強い |

## 10. まとめ

### 全体の流れ

$$\text{アイデア} \xrightarrow{\text{HDL/ライブラリ/コンパイラ}} \text{R1CS} \xrightarrow{\text{証明システム}} \pi \text{（ZK証明）}$$

### 3つのアプローチ

```
                    アイデア
                   /   |   \
                  /    |    \
               HDL  ライブラリ  PL
            (circom)(arkworks)(noir)
                  \    |    /
                   \   |   /
                    R1CS
                      |
                   ZK証明（π）
```

### 最終比較表

| 特性 | HDL (Circom) | ライブラリ (Arkworks) | PL (ZoKrates/Noir) |
|------|-------------|---------------------|-------------------|
| 学習難度 | 高 | 中 | 低 |
| 制約の透明性 | 高 | 高 | 低 |
| Witness計算の柔軟性 | 高 | 最高 | 低〜中 |
| 既存言語との親和性 | 低 | 高（Rust） | 中 |
| 最適化の自動化 | 低 | 低 | 高 |
| 実績・エコシステム | 大（Zcash等） | 大 | 成長中 |

### DeIn（分散型保険）への応用観点

ZKPをDeInのような分散型保険プロトコルに応用する場合、以下の用途が考えられます：

1. **プライベートな損害証明**: 実際の損害データを開示せずに保険金請求の正当性を証明
2. **KYC/AML証明**: 個人情報を開示せずにKYCを通過していることを証明
3. **パラメトリック条件の検証**: 地震の規模データが閾値を超えていることをオンチェーンで証明

**ツール選択の観点:**
- **プロトタイプ段階**: ZoKrates（学習コスト低）
- **本番実装**: Circom（Ethereum/Groth16との親和性高）またはArkworks（Rustエコシステム）
- **Circomの利点**: snarkjs との組み合わせでEthereumのSolidityコントラクトで検証可能
