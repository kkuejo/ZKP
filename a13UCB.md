## 第1章：ゼロ知識証明とは何か（前提知識）

### ゼロ知識証明の直感的説明

「私はこの秘密を知っている」ということを、**秘密そのものを見せずに証明する**技術です。

**例え話：** アリスが洞窟の中の秘密の合言葉を知っているとします。ボブは「本当に知っているの？」と疑っています。ゼロ知識証明を使えば、アリスは合言葉を一切口にせず、「確かに知っている」とボブを納得させることができます。

### SNARKとSTARK

この講義で使われる主な暗号ツールです：

- **SNARK（Succinct Non-interactive ARgument of Knowledge）**：簡潔な非対話型知識証明
  - 証明サイズが小さい
  - 検証が速い
  - 「プルーバー（証明者）」が証明を作り、「ベリファイヤー（検証者）」が確認する

数学的には、回路 $C$ と入力 $x$、秘密の証人 $w$ があるとき、

$$C(x, w) = 1$$

を満たす $w$ を知っていることを、$w$ を明かさずに証明します。

### 本講義が選んだSNARK方式（Slide 4）

```
Sumcheck IOP + KZG commitment scheme
（例：Hyperplonk, Honk）
```

- **Sumcheckプロトコル**：多変数多項式の和を効率的に検証する手法
- **KZGコミットメント**：Kate-Zaverucha-Goldberg。多項式を1つの楕円曲線上の点にまとめる技術

再帰的証明のために **Halo2スタイルの曲線サイクル** を使います。これにより、「証明の証明の証明…」を効率的に行えます。

## 第2章：ブロックチェーンの仕組みを再確認（Slides 5–6）

### ブロックチェーン＝状態遷移機械

ブロックチェーンは**状態遷移機械（State Machine）**として理解できます：

$$S_{i+1} = \text{transition}(S_i, \text{TX}_i)$$

- $S_i$：ブロック $i$ 時点の「世界の状態」（誰がいくら持っているか等）
- $\text{TX}_i$：トランザクション（取引）
- $S_{i+1}$：取引後の新しい状態

**Ethereumの例：** 全アカウントの残高・コード・ストレージがまとめて「状態」を構成しています。

### プライベートな状態遷移機械とは？

通常のブロックチェーンでは状態（残高・取引内容）が**全員に見えます**。プライベートな状態遷移機械では、これを暗号化し、**当事者だけが内容を知れる**ようにします。

## 第3章：プライベート状態の実現方法（Slides 7–9）

### 問題点：状態の更新がプライバシーを漏らす

ブロックチェーンの状態はMerkleツリーで管理されます。

**Merkleツリーとは：** データの塊を木構造でハッシュ化したもの。ルートハッシュ1つで全データの整合性を証明できます。

$$\text{Root} = H(H(H(d_1, d_2), H(d_3, d_4)), \ldots)$$

ここで $H$ はSHA-256などのハッシュ関数。

**問題：** もしMerkleツリーのエントリを「修正」「削除」すると、ツリーの構造変化から「どのデータが変わったか」が外部に漏れてしまいます。

**解決策：** Merkleツリーを**追記専用（Append-only）**にする。

### Nullifier（無効化器）の概念（Slide 8）

状態の「削除」を表現するために **Nullifier（ヌリファイヤー）** を使います：

$$\text{Nullifier} = \text{Enc}(\text{UTXO},\ \text{owner.sk})$$

- **UTXO（Unspent Transaction Output）**：まだ使われていない状態のかたまり
- **owner.sk**：所有者の秘密鍵（Secret Key）
- **Enc**：暗号化関数

**仕組み：**
1. 状態（UTXO）は暗号化されてMerkleツリーに追記される
2. 状態を「削除」したいとき、ヌリファイヤーを**別のセット（Nullifier Set）**に追記する
3. 「状態が有効か？」の確認は「ヌリファイヤーセットにないか？」をチェックするだけ

**プライバシーが守られる理由：** ヌリファイヤーは秘密鍵なしには元のUTXOと紐づけられないため、外部の観察者にはどの状態が無効化されたかわかりません。

### 具体例（Slide 9）

```
data  = "we hold these truths to be self-evident"
owner = bfnklyn.eth（所有者のアドレス）

UTXO      = Enc(data, owner, owner.sk)
             → 0xa9ef... としてMerkleツリーへ

Nullifier = Enc(UTXO, owner.sk)
             → 0xe12c... としてNullifier Setへ
```

これにより **UTXOとNullifierを結びつけられるのは秘密鍵の持ち主だけ** です。

### 結論：プライベート状態はUTXO構造を持つ

このアーキテクチャは必然的に **UTXOモデル**（Bitcoinに似た構造）になります：

- 状態を「作る」＝新しいUTXOをMerkleツリーに追記
- 状態を「消す」＝ヌリファイヤーをNullifier Setに追記
- 状態を「更新する」＝古いUTXOを消して新しいUTXOを作る

## 第4章：UTXOだけでは足りない（Slides 10–12）

### 問題1：レースコンディション（競合状態）

1つのUTXOを、同じブロック内で**2回変更しようとする競合**が起きます。

**例え話：** 銀行の同じ口座から、2人が同時にATMで引き出そうとする状況。UTXOモデルでは1つのUTXOは1ブロックで1回しか使えません。

### 問題2：所有権なしの状態更新

**担保の強制清算**を例に考えます：DeFi（分散型金融）では、担保価値が下がったとき、誰か（清算者）が強制的に担保を売却します。しかしUTXOモデルでは、状態を更新するには**秘密鍵が必要**なので、第三者が自動で清算できません。

### 解決策：プライベート状態 ＋ パブリック状態の組み合わせ（Slide 12）

| 状態の種類 | 特徴 | 実行者 |
|-----------|------|--------|
| プライベート状態遷移 | ユーザーが証明を生成 | ユーザー本人（クライアント側） |
| パブリック状態遷移 | 順番通りに実行 | マイナー（Eth1）・バリデータ（Eth2）・シーケンサー（L2） |

## 第5章：プライベート＋パブリック状態を持つスマートコントラクト（Slides 13–18）

### コントラクトの構造（Slide 15–16）

スマートコントラクトは**プライベート関数**と**パブリック関数**から構成されます：

**プライベート関数ができること：**
- UTXOツリーの更新
- Nullifierセットの更新
- 過去のパブリック状態の読み取り（履歴参照のみ）
- パブリック関数の一方向呼び出し（戻り値なし）

**パブリック関数ができること：**
- UTXOツリーの更新
- Nullifierセットの更新
- パブリック状態の読み書き

### プロトコル表現（Slide 17）

コントラクトの「関数」は **ZK SNARKの検証鍵（Verification Key, vk）** で定義されます：

- 関数 $f$ は検証鍵 $vk_f$ を持つ
- コントラクト = 関数の検証鍵の集合 $\{vk_{f_1}, vk_{f_2}, \ldots\}$
- 回路の公開入力が統一されたABI（Application Binary Interface）に従う

### スマートコントラクトのABI例（Slide 18）

| 公開入力の範囲 | 用途 |
|--------------|------|
| 0–9 | 関数の引数パラメータ |
| 10 | UTXOツリーの状態ルート |
| 11 | Nullifierツリーの状態ルート |
| 12 | パブリックツリーの状態ルート |
| 13 | msg.sender（暗号化済み） |
| 14–19 | 追加するUTXOリーフ |
| 20–29 | 追加するNullifierリーフ |
| 30–39 | イベントパラメータ |

## 第6章：プライベートカーネル回路（Slides 20–33）

この章が本講義の核心です。

### カーネルとは何か（Slides 21–22）

**一般的なOSにおけるカーネル：**
- ユーザーコードとCPU・ハードウェアの間のソフトウェア層
- コード実行ルールを強制する
- リソースアクセスとアプリ間通信を管理する

**ZK SNARKにおけるカーネル：**
- ユーザーコード（Noirで書かれたコントラクト等）とプロトコル実行層（L2ロールアップ等）の間の回路層
- コードのデプロイ・実行ルールを強制する
- コントラクト内のデータ・関数へのアクセスを管理する
- 一部の情報のプライバシーを維持する

### プライベートカーネル回路が必要な理由（Slides 23–25）

#### 理由1：プライバシー保護
- ユーザーのIDを明かさずに認証する
- 呼び出されているコントラクトを隠す

#### 理由2：コンポーザビリティ（組み合わせ可能性）
- 関数が他のコントラクトの関数を呼び出せる必要がある
- 各コントラクト関数はそれぞれ**独自の回路と証明**を持つ

#### 理由3：プライベートな関数間呼び出し（Slide 25）

次のような状況を考えます：

```
function B(some_secret) {
    // 秘密を使い、新しい秘密を返す
    return some_secret + other_secret;
}

function A(some_secret) {
    // Bを呼び出し、秘密を渡す
    new_secret = B(some_secret);
    // maybe call C...
}

// アリスは A(12345) を呼び出す
// "12345" は重要な秘密！
```

`A.foo()` → `B.bar()` の呼び出しで**秘密の情報が渡される**場合、その情報をシーケンサー等の第三者に見せずに証明を結合する必要があります。

### プライベートカーネルの概要（Slides 26–27）

カーネル回路は以下を行います：

1. **1つのプライベート関数呼び出し**の正しい実行を検証する回路
2. 構造は**再帰的（Recursive）**

再帰的とはどういう意味か？

$$\pi_{\text{kernel}}^{(n)} = \text{Prove}\left(\pi_{\text{kernel}}^{(n-1)},\ \pi_{\text{function}}^{(n)}\right)$$

つまり、「前のカーネル証明」を検証しながら「次の関数証明」を組み込む、という操作を繰り返すことで、コールスタック全体を1つの証明にまとめられます。

**ユーザーが送信するもの：**
- コールスタック全体の実行に対する**単一の証明**
- 以下のプライバシーが保護される：
  - ユーザーのアイデンティティ（`tx.origin`）
  - ネストされた関数の引数・戻り値
  - 状態の読み取り内容
  - 呼び出された関数そのもの

### カーネルが各関数呼び出しで証明すること（Slide 28）

- 署名済みTXリクエストがコールスタックの最初の呼び出しと一致する
- 関数が**関数ツリー**に存在する
- コントラクトが**コントラクトツリー**に存在する
- 関数が参照するコミットメントが**データツリー**に存在する
- 新しいコミットメント・Nullifier・コントラクトを収集する
- 前のカーネル証明を検証する
- 現在処理中の関数の証明を検証する

### カーネルへの入力（Slide 29）

```
type PrivateKernelInputs = {
    signed_tx_request: SignedTxRequest,   // ユーザーの最初の要求
    previous_kernel: PreviousKernelData,  // 前のカーネルの蓄積データ
    private_call: PrivateCallData,        // 現在の関数呼び出しデータ
}
```

`TxRequest` には次が含まれます：
- `from`（送信者アドレス）
- `to`（コントラクトアドレス）
- `function_data`（呼び出す関数）
- `args`（引数、8要素の配列）
- `nonce`（プライベートはランダム、パブリックはインクリメンタル）
- `chain_id`（チェーンID = 1234など）

### カーネルの再帰構造（Slides 30–31）

**再帰の流れ（Slide 30）：**

```
ECDSA署名
    ↓
プライベートカーネル回路1
    ├── 前のカーネル証明を検証
    └── 関数回路1の証明を検証
    ↓ 証明
プライベートカーネル回路2
    ├── カーネル証明1を検証
    └── 関数回路2の証明を検証
    ↓ 証明
プライベートカーネル回路3
    ├── カーネル証明2を検証
    └── 関数回路3の証明を検証
    ↓
最終プライベートカーネルSNARK（1つの証明）
```

**コールスタックを通じた再帰（Slide 31）：**

```solidity
// Contract1.sol
import Contract2;
contract Contract1 {
    private uint x;
    function1(uint a, uint b, uint c) {
        d = Contract2.function2_1(a, b);  // 別コントラクトを呼ぶ
        x += d;
        Contract2.function2_3(c, x);
    }
}
```

このような**跨コントラクト呼び出し**が再帰的カーネルで安全に処理されます。

### プライベートカーネル回路のアーキテクチャ（Slide 32）

フローチャートで表すと：

```
入力パラメータ：
  - TXシグネチャ？
  - カーネル証明？
  - コール深度
  - プライベートコールスタック
  - パブリックコールスタック
  - UTXOキュー
  - Nullifierキュー
  - "Oracle状態"

↓
コール深度 == 0？
  ├─ YES → msg.senderのTXシグネチャを検証
  └─ NO  → 前のカーネル証明を検証
              ↓
            前の出力パラメータ == 現在の入力パラメータを検証
↓
プライベートコールスタックからfをpop
↓
f.vkが存在することを確認
f.vkで証明を検証
↓
f.proofから公開入力pを抽出
↓
├─ pがプライベート呼び出しを含む → プライベートコールスタックへpush
├─ pがパブリック呼び出しを含む  → パブリックコールスタックへpush
└─ pがUTXO/Nullifier更新を含む → UTXO/Nullifierキューへpush

↓
出力パラメータ：
  - プライベートコールスタック
  - パブリックコールスタック
  - UTXO/Nullifierキュー
  - 入力Oracle状態
```

### カーネルが行わないこと（Slide 33）

| やらないこと | 誰がやるか |
|------------|----------|
| 関数回路の実行 | カーネルより前（ユーザーが実行） |
| ツリーへの挿入（コミットメント・Nullifier等） | ロールアップ回路（シーケンサー・プルーバー） |
| 別TXのマージ | ロールアップ回路（シーケンサー・プルーバー） |

## 第7章：パブリックカーネル回路（Slides 34–38）

### メンプール内でのTXの状態（Slide 35）

ユーザーがTXを送信すると：
- プライベートカーネルのZK証明が付属
- **プライベートコールスタックは空**（すべて処理済み）
- **パブリックコールスタックには実行待ちの関数**が含まれる

これらのパブリック関数を**パブリックカーネル回路**で検証する必要があります。パブリックカーネル証明は**シーケンサー・プルーバー**が生成します。

### パブリック関数証明の計算（Slide 36）

- パブリック関数証明は**第三者のシーケンサー・プルーバー**が計算する
- 関数証明はパブリックカーネル回路でラップされる

**大きな問題点：** シーケンサーは計算の対価として公正に報酬を受け取る必要がある

### ユーザーとシーケンサーの信頼問題（Slide 37）

関数証明が無効になるケースは2つ：

1. **公開入力の選択が制約を満たさない**（= TXがエラーを起こす）→ TX送信者の責任
2. **ウィットネスの割り当てが意図的に無効**→ シーケンサーの責任

| 失敗の原因 | 責任者 |
|----------|--------|
| TXがエラー（実行不可） | TX送信者 |
| 証明が意図的に無効 | シーケンサー |

### 解決策：VM（仮想マシン）が必要（Slide 38）

- **有効なVM証明** = シーケンサーはユーザーをいじめられない
- **VM証明の実行結果** = パブリック入力として返却できる → ユーザーはシーケンサーに無償労働させられない

$$\text{valid\_tx} \iff \text{valid\_VM\_proof}$$

## 第8章：仮想マシンの仕組み（Slides 39–44）

### CPUアーキテクチャの基礎（Slide 40）

| 概念 | 意味 |
|------|------|
| **オペコード（Opcode）** | CPUの命令セットの一部。原子的な操作として扱われる |
| **マイクロコード（Microcode）** | オペコードを細かい操作に分解したもの。1クロックで1マイクロコード操作が実行される |
| **レジスタ（Registers）** | 処理中のデータを格納する高速メモリ |
| **RAM** | 残りのデータを格納するメモリ |
| **ALU（算術論理演算装置）** | 算術命令を実行するハードウェア |

### SNARK VMの仕組み（Slides 42–44）

SNARK VMは**ZK証明できる仮想マシン**です。コンピュータの実行を証明として記録します。

#### 回路の構造（Slide 42）

1列 = 1つの多項式コミットメント
1行 = 1つのゲート（演算ステップ）

```
実行時カラム（Proverがコミット）：
  PC（プログラムカウンタ）
  OP（オペコード）
  レジスタ
  ゲートセレクタ
  メモリテーブル
  MC（マイクロコード）

ルックアップテーブル：
  プログラム固有（事前計算）
  VM固有（事前計算）
```

#### VMオペコードの例（Slide 44）

| オペコード | マイクロコード数 | ゲート式 | 手法 |
|----------|--------------|--------|------|
| Add | 1 | $R1_{i+1} = R1_i + R2_i$ | カスタムゲート |
| MOV [R1] | 1 | $R1_{i+1} = M[R1_i]$ | ルックアップ |
| XOR R1 [R2] | 1 | $R1_{i+1} = R1 \oplus M[R2_i]$ | カスタムゲート＋ルックアップ |
| SHA256 | 3,000 | $M[R1_i] = \text{SHA256}(M[R2_i])$ | 3,000ゲート＋ルックアップ |
| JUMPI X | 1 | $PC_{i+1} = (R1_i == 0)\ ?\ PC_i + 1 : X$ | カスタムゲート |

**注目点：** SHA256は3,000マイクロコード操作を必要とします。これがZK VMの計算コストが高い理由の一つです。

## 第9章：ロールアップ回路（Slides 45–50）

### なぜロールアップが必要か（Slide 46）

ブロック内の全TXの検証は**ベリファイヤーのコストが高い**ため問題があります。

理想：コンセンサス層（バリデータ等）は**ブロックの正しさの証明のみを検証**すればよい。

$$\text{コスト} \propto O(1) \quad \text{（ブロック内TX数によらず一定）}$$

### ベースロールアップ回路（Slide 47）

ベースロールアップは**2つのTX証明**を1つにまとめます：

入力：
- $\pi_1$, $\pi_2$：2つのパブリックカーネル証明
- $(P_a, P_b)$：各証明のペアリングポイント（KZGコミットメント用）
- $(\vec{n}, \vec{c}, \vec{d})$：Nullifier, コミットメント, データ
- $(T_{\text{old}}, T_{\text{bnew}})$：古いツリーと新しいバッファツリー

処理：
1. $vk_{\text{kernel}}$ で2つの証明を検証＆集約
2. Merkle証明でツリー更新を検証
3. SHA256で公開入力ハッシュ $H_{\text{PI}}$ を計算

### マージロールアップ回路（Slide 48）

マージロールアップは**2つのベースロールアップ証明**をさらにまとめます：

$$\pi_{\text{merge}} = \text{Prove}\left(\pi_{\text{base}_1},\ \pi_{\text{base}_2},\ vk_{\text{base}}\right)$$

SHA256で2つの $H_{\text{PI}}$ を結合：

$$H_{\text{PI}}^{\text{merge}} = \text{SHA256}(H_{\text{PI}}^{(1)},\ H_{\text{PI}}^{(2)})$$

### ロールアップツリー構造（Slide 49）

```
                    マージロールアップ（ルート）
                   /                    \
          マージロールアップ          マージロールアップ
          /          \               /          \
    ベースロールアップ ベースロールアップ ベースロールアップ ベースロールアップ
    /    \            /    \          /    \        /    \
   π₁    π₂         π₃    π₄       π₅    π₆      π₇    π₈
```

各回路は2証明/回路でロールアップ → **小さな回路サイズ = 速い証明 = 分散化を助ける**

### ルートロールアップ回路（Slide 50）

最終的に**証明システムを段階的に変換**して検証コストを下げます：

$$\text{Honk} \rightarrow \text{Ultra} \rightarrow \text{Std} \rightarrow \pi_{\text{final}}$$

- **Honk**：再帰証明に最適化された高効率SNARK
- **Ultra**：中間形式
- **Std**：標準的なSNARK（Ethereum上で安価に検証可能）

最終証明 $\pi_{\text{final}}$ はEthereum上の `RollupProcessor` スマートコントラクトに送られ、`dataRoot`, `nullRoot`, `rootRoot` の3つのMerkleルートを更新します。

## 第10章：全体のまとめ（Slides 52–53）

### 状態の種類（Slide 52）

| 状態ツリー | 内容 |
|----------|------|
| **プライベート状態ツリー** | 暗号化されたUTXO（Merkle追記専用） |
| **パブリック状態ツリー** | 公開された状態（読み書き可能） |
| **コントラクト状態ツリー** | デプロイ済みコントラクトの検証鍵集合 |
| **Nullifierセット** | 無効化された状態の記録 |

### 回路の種類と役割（Slide 53）

| 回路 | 役割 |
|------|------|
| **プライベートカーネル回路** | プライベート関数の実行を検証 |
| **パブリックカーネル回路** | パブリック関数の実行 ＋ プライベートカーネル証明を検証 |
| **ロールアップ回路** | パブリックカーネル証明の検証 ＋ 状態更新の実行 |
| **ルートロールアップ回路** | ロールアップ証明の検証（低コストSNARKプロトコルを使用） |

### TX処理の全体フロー

```
[ユーザー（クライアント側）]
1. プライベート関数を実行
2. 各関数の証明（π）を生成
3. プライベートカーネル回路を再帰的に適用
4. 最終プライベートカーネル証明を生成
5. TXをメンプールに送信

[シーケンサー・プルーバー]
6. パブリックコールスタックの関数をVM上で実行
7. パブリックカーネル回路で各実行を証明
8. ベースロールアップ回路でTX証明を2つずつ集約
9. マージロールアップ回路で再帰的に集約
10. ルートロールアップ回路でπ_finalを生成

[コンセンサス層（L1 Ethereum等）]
11. π_finalを検証（1回の検証でブロック全体を確認）
12. 状態ルートを更新
```

## DeInへの応用：参考メモ

この講義の内容はAztec Networkのアーキテクチャですが、**DeIn（分散型パラメトリック地震保険）**との関連で考えると：

| Aztecの概念 | DeInでの対応 |
|-----------|------------|
| プライベート状態（UTXO） | 保険証書（NFT/ERC-721）の所有情報 |
| Nullifier | 保険金支払い後の証書無効化 |
| パブリック状態 | オラクル（JMA地震データ）、プール残高 |
| スマートコントラクト | 自動支払いロジック、DAOガバナンス |
| ZK証明 | 保険請求の正当性証明（将来的な応用） |

現在のDeInはEthereumのパブリック状態のみを使用していますが、将来的にAztecのようなプライバシー層を追加することで、**保険加入者の個人情報（位置情報・資産情報）を秘匿しながら自動支払い**を実現できます。

## 用語集

| 用語 | 意味 |
|------|------|
| **ZKP（Zero-Knowledge Proof）** | 秘密を明かさずに知識の所有を証明する技術 |
| **SNARK** | 簡潔な非対話型知識証明。証明が小さく検証が速い |
| **UTXO** | Unspent Transaction Output。未使用の取引出力 |
| **Nullifier** | UTXOを無効化するための暗号トークン |
| **Merkleツリー** | データをハッシュ関数で木構造に整理した認証構造 |
| **カーネル回路** | OSのカーネルに相当するZK回路。実行ルールを強制する |
| **ロールアップ** | 多数のTXを1つの証明にまとめてL1のコストを削減する技術 |
| **シーケンサー** | L2でTXの順序付けと証明生成を担当するオペレータ |
| **KZGコミットメント** | 多項式を1点にまとめる暗号コミットメントスキーム |
| **再帰証明（Recursive Proof）** | 証明の正しさ自体をさらに別の証明で示す手法 |
| **vk（Verification Key）** | ZK証明の検証に使う公開鍵 |
| **Noir** | Aztecが開発したZK回路記述言語 |
| **Honk** | Aztecが開発したSNARKプロトコル（再帰に最適化） |
