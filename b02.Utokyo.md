## 1. まず「ゼロ知識証明（ZK）」とは何か？

### 直感的な説明

「ゼロ知識証明」とは、**「ある事実を知っている」ことを、その事実の中身を一切明かさずに証明できる暗号技術**です。

**日常の例え話：**
- あなたが「私はパスワードを知っている」と証明したい。
- でも実際のパスワードを相手に教えたくない。
- ゼロ知識証明を使えば、「知っている」という事実だけを、パスワードを明かさずに証明できます。

### ZKの3つの性質

| 性質 | 意味 |
|------|------|
| **完全性（Completeness）** | 正直な証明者は、正しい主張を必ず証明できる |
| **健全性（Soundness）** | 嘘をついている人は証明を偽造できない |
| **ゼロ知識性（Zero-Knowledge）** | 検証者は「事実が正しい」以外の情報を何も得られない |

### ZK回路とは？

ZKでは、証明したい計算を「**回路（Circuit）**」として記述します。

- **Public Input（公開入力）**：誰でも見られる入力値
- **Private Input（秘密入力）**：証明者だけが知る入力値（外部には公開されない）
- **Output（出力）**：計算結果（公開される）
- **Proof（証明）**：「正しく計算した」という証拠データ

$$
\text{ZK回路}: (\text{Public Input}, \text{Private Input}) \rightarrow (\text{Output}, \text{Proof})
$$

検証者は Public Input・Output・Proof の3つだけ受け取り、Private Inputを知らなくても「計算が正しい」ことを確認できます。

## 2. ZKがブロックチェーンの何を補うのか？

### ブロックチェーンの2つの弱点

#### 弱点①：透明性（プライバシーがない）
- ブロックチェーンに記録されたすべてのトランザクション（送金記録）は**全世界に公開**されます。
- 誰が誰にいくら送ったか、全員がわかってしまう。

#### 弱点②：計算能力の限界
- スマートコントラクト（ブロックチェーン上で動くプログラム）は計算できるが、**複雑な計算はコストが高く、現実的でない**。
- 計算が複雑になればなるほど、全ノードが同じ計算を繰り返す必要があり、非効率。

### ZKが補う2つのこと

| ブロックチェーンの弱点 | ZKによる解決 |
|------------------------|--------------|
| 透明性（プライバシーなし） | **入力の秘匿化**：Private Inputを使って、重要な情報を隠しながら計算できる |
| 計算能力の限界 | **検証コストの簡略化**：どんな複雑な計算でも、その「証明（Proof）」の検証コストは一定 |

#### 「検証コストが一定」とはどういう意味か？

通常のブロックチェーンでは、計算が複雑なほど検証コスト（ガス代）が高くなります。

一方ZKでは：

$$
\text{検証コスト} = \text{一定} \quad (\text{計算の複雑さに依存しない})
$$

例えば、1000個のトランザクションを一括処理しても、その「証明」を検証するコストはシンプルな1トランザクションとほぼ同じです。

#### 注意点（限界）
- ZKは「回路への入力値の一部を隠す」だけ。任意の情報を自由に隠蔽・保存できるわけではない。
- オフチェーン（ブロックチェーン外）での証明生成に大きな計算コストと時間がかかる。

## 3. 事例①：Zcash（入力の秘匿化）

### 3-1. Zcashとは？

**Zcash**は2016年にリリースされた**プライバシー重視の暗号通貨**です。

- ビットコインと同じブロックチェーン技術を使っているが、送金情報を隠せる。
- ZKを採用している。
- 通常の公開トランザクションと、秘匿化された「**シールドトランザクション**」の両方が使える。

### 3-2. なぜビットコインだけではダメなのか？

「ビットコインも匿名じゃないの？」と思うかもしれません。

**ビットコインの匿名性の限界：**
- アドレス自体は匿名（本名は載っていない）。
- しかし、**すべての送金履歴がブロックチェーンに公開**されている。
- お店でビットコイン払いをしたとき、お店は「その客のアドレス」と「あなた」を紐付けられる。
- 一度紐付けられると、**過去の全送金履歴**が追跡可能になる。

**Zcashが実現する2種類のプライバシー：**

| 種類 | 意味 |
|------|------|
| **Confidential（機密性）** | 送金金額が秘匿化されている |
| **Anonymous（匿名性）** | 送金者・送金先アドレスが秘匿化されている |

Zcashはこの両方を同時に実現しています。

### 3-3. UTXOモデルの復習

Zcashはビットコインと同じ「**UTXOモデル**」を使っています。

**UTXOとは？**（Unspent Transaction Output：未使用トランザクション出力）

お金の管理をコインで考えると分かりやすい：

- 「500円玉1枚」がUTXOの1単位（お釣りのあるコイン）
- 送金するには、手持ちのコインを「使う（消費する）」
- お釣りが出るなら、新しいコインとして自分に戻ってくる

**ビットコインのUTXOの仕組み：**

```
【前のトランザクション】         【今のトランザクション】
  Vout（出力）                    Vin（入力）
  ┌─────────────┐               ┌─────────────────┐
  │ ロッキング  │ ──────────▶  │ アンロッキング  │ ──▶ Vout（出力1）
  │ スクリプト  │               │ スクリプト      │ ──▶ Vout（出力2）
  └─────────────┘               └─────────────────┘
```

- **ロッキングスクリプト**：「このコインを使えるのは、○○の秘密鍵を持つ人だけ」という鍵
- **アンロッキングスクリプト**：「私が秘密鍵を持っている」という証明（署名を含む）
- インプットの合計＝アウトプットの合計になることで、不正な送金（お金の水増し）を防止

### 3-4. Zcashの全体像

Zcashのシールドトランザクションは大きく2つの仕組みで成り立っています：

#### ① 送金情報の暗号化（機密性・匿名性の実現）

送金の内容（誰に、いくら）を「**Note（ノート）**」と呼ばれるデータにまとめ、暗号化してブロックチェーンに記録します。

Noteの中身：
- 送金先の公開鍵
- 送金金額
- $\rho$（ランダムな値、後述のNullifierに使用）

このNoteを暗号化したものがブロックチェーンに記録されるため、第三者には中身が見えません。

#### ② ZKプルーフによる正当性の証明

「暗号化された正しい送金を行った」ことをZKで証明します。トランザクションにZKプルーフが含まれています。

#### ③ Nullifierによる二重支払い防止（後述）

### 3-5. 暗号化と鍵管理（DH鍵共有）

送金先が暗号化されたNoteを復号するには、送金者の鍵情報が必要です。Zcashでは**Diffie-Hellman（DH）鍵共有**という手法を使います。

**DH鍵共有とは？**

AliceとBobが、盗聴される可能性のある通信路で「共通の秘密鍵」を安全に共有する方法です。

数学的には、以下の性質を利用します：

$$
\text{Alice の秘密鍵} \times \text{Bob の公開鍵} = \text{Bob の秘密鍵} \times \text{Alice の公開鍵} = \text{共通鍵}
$$

具体的には（離散対数問題ベース）：

$$
g^{ab} \equiv g^{ba} \pmod{p}
$$

- $g$：公開のベース（生成元）
- $p$：公開の大きな素数
- $a$：Aliceの秘密鍵、$g^a$：Aliceの公開鍵
- $b$：Bobの秘密鍵、$g^b$：Bobの公開鍵
- 共通鍵 = $g^{ab} \pmod{p}$（第三者は$a$も$b$も知らないので計算できない）

**Zcashでの送金フロー：**

```
送金者                                    送金先
  │                                          │
  │ ①シールドアドレス（公開鍵を含む）を受け取る ◀───│
  │                                          │
  │ ②ephemeralKey（一時鍵）を生成             │
  │   （送金ごとに新しい鍵ペアを生成）          │
  │                                          │
  │ ③DH鍵共有で共通鍵を生成し、Noteを暗号化   │
  │                                          │
  │ ④暗号化NoteとephemeralKey（公開鍵）をTxに  │
  │                                          │
  │ ────────────── Tx送信 ────────────────▶  │
  │                                          │
  │               ⑤送金先は自分の秘密鍵と     │
  │                 ephemeralKeyで共通鍵生成  │
  │                 → Noteを復号化            │
```

暗号化アルゴリズムとして `AEAD_CHACHA20_POLY1305` が使われています。

> **ポイント：** シールドアドレスは送金のたびに新規生成します。同じアドレスを使い回すと、複数の送金が同一人物によるものだと追跡されやすくなるためです。

### 3-6. NullifierとNoteコミットメント（二重支払い防止）

ZKで送金の正当性は証明できますが、「同じNoteを2回使う（二重支払い）」を防ぐ仕組みが別途必要です。

#### コミットメントとNullifierの関係

```
                  SHA256
Note ────────────────────────▶ Noteコミットメント（公開）
│                                （Noteの中身のハッシュ値）
│ + 送金先の秘密鍵
│
▼
Nullifier（公開）
（ρと秘密鍵から擬似乱数で生成、決定論的）
```

**コミットメント**とは：Noteのデータをハッシュ関数にかけた値。元のデータを知らないと生成できないが、コミットメントからは元データを復元できない（一方向性）。

$$
\text{Commitment} = \text{SHA256}(\text{Note})
$$

$$
\text{Nullifier} = \text{PRF}(\rho, \text{秘密鍵})
$$

（PRF：擬似乱数関数）

#### 二重支払いをどう防ぐか？

| 仕組み | 役割 |
|--------|------|
| **Noteコミットメント** | 「このNoteが存在する」ことの公開証拠。マークルツリーで管理 |
| **Nullifier** | 「このNoteを使った」という公開フラグ |

**フロー：**
1. 送金が行われると、Nullifierがブロックチェーンに公開される。
2. 次に同じNoteを使おうとすると、同じNullifierが生成される（決定論的）。
3. すでに公開済みのNullifierを含むTxは**無効**とみなされる。

#### なぜNullifierという別の識別子が必要なのか？

「使用済みフラグをコミットメント自体でやればよいのでは？」という疑問が生じますが：

- コミットメントで使用済み管理をすると、「このコミットメントは使用済み」という情報が公開される。
- → どのNoteが使われたか追跡できてしまい、匿名性が破れる。

Nullifierは：

$$
\text{コミットメント} \xrightarrow{\times} \text{Nullifier} \quad (\text{逆算不可能})
$$

$$
\text{Nullifier} \xrightarrow{\times} \text{コミットメント} \quad (\text{逆算不可能})
$$

両方向に逆算不可能なため、NullifierからどのNoteが使われたか追跡できません。

> **ポイント：** NullifierとコミットメントのリンクはZK回路内でのみ検証され、外部には公開されません。小規模なシステムでも、消去法的に誰のTxかを推測されにくくなっています。

### 3-7. ZKプルーフで何を証明するか？

送金時にZK回路に入力される情報と、証明される内容：

**Public Input（公開される入力）：**
- 入力NoteのNullifier（使用済みフラグとして公開）
- 出力NoteのコミットメントMerkle tree root（$r_t$）
- 入力NoteコミットメントのMerkle path

**Private Input（秘密の入力）：**
- 自分の秘密鍵
- 入力Note（復号して取得）
- 出力Note（新たに生成）

**ZK回路が証明すること：**

| チェック内容 | 意味 |
|-------------|------|
| 秘密鍵と公開鍵のペアが正しい | 本人からの送金かチェック |
| 入力Noteのコミットメントがマークルツリーに存在する | 正当なNoteを使っているかチェック |
| Nullifierが正しく計算されている | Nullifierの正当性チェック |
| 入力の送金額と出力の送金額が一致 | お金の水増しがないかチェック |

これらすべてをPrivate Inputを公開せずに証明できます。

### 3-8. ブロックとトランザクションの構造

ブロックチェーンに記録される情報（第三者が見られる情報）：

**ブロック：**
- 前ブロックハッシュ
- PoWのナンス・ターゲット
- Txルート
- **Noteコミットメントルート**（全NoteコミットメントのMerkle root）

**トランザクション（シールド）：**
- 入力NoteのNullifier（複数）
- 出力NoteのコミットメントとMerkle root
- 送金が正しいというZKプルーフ
- 暗号化された出力Note
- EphemeralKey（送金者の一時公開鍵）
- 署名（改ざん耐性のため。Tx毎に乱数で鍵生成）

> 第三者はこれらの情報から、「誰が誰に送ったか」「いくら送ったか」を一切知ることができません。

### 3-9. Zcashまとめ

```
送金内容の秘匿
  ↳ Confidential（金額を隠す）+ Anonymous（送金者・先を隠す）
  ↳ DH鍵共有 + CHACHA20暗号化でNoteを暗号化

正当性の証明
  ↳ ZKプルーフで、「正しい送金者・正しい金額」を秘密鍵非公開で証明

二重支払い防止
  ↳ Commitment（公開）+ Nullifier（使用時に公開）の2つの識別子で実現
  ↳ 両者の関連はZK回路内でのみ確認、外部への漏洩なし
```

## 4. 事例②：zkRollup / zkEVM（検証コストの簡略化）

### 4-1. そもそも「Rollup（ロールアップ）」とは？

Ethereumはメインチェーン（L1）の処理速度に限界があります（秒間15〜30トランザクション程度）。

**Rollup**はこれを解決するスケーリング技術です：

```
【L1（Ethereumメインチェーン）】
  Block1         Block2         Block3
  Header         Header         Header
  ┌─────┐       ┌─────┐       ┌─────┐
  │ステ│       │ステ│       │ステ│
  │ート│       │ート│       │ート│
  │ルート│     │ルート│     │ルート│
  └──┬──┘       └──┬──┘       └──┬──┘
     │               ▲             │
     │       L2のステートルートをL1に記録
     ▼               │
【L2（Rollupチェーン）】
  多数のL2トランザクションをバッチ処理
  L2Txデータ → L1にcalldataとして書き込む
```

L2でたくさんのTxをまとめて処理し、その結果だけをL1に記録することで、コストと速度を改善します。

### 4-2. Optimistic RollupとzkRollupの違い

| | Optimistic Rollup | zkRollup |
|--|-------------------|----------|
| **考え方** | 「正しいだろう」と楽観的に受け入れる | ZKで数学的に正しさを証明する |
| **検証方法** | 一定期間（7日）チャレンジ期間を設ける | ZKプルーフで即時検証 |
| **セキュリティ** | 誰かが不正を見つけたら異議申し立て | プルーフが通れば絶対に正しい |
| **最終確定** | 7日後 | 即時 |
| **計算コスト** | 低い | 高い（プルーフ生成） |

zkRollupでは：

$$
\text{L2で行った全計算} \xrightarrow{\text{ZK証明}} \text{Proof（小さなデータ）} \xrightarrow{\text{L1で検証}} \text{OK/NG}
$$

どんなに複雑な計算でも、Proofの検証コストは一定です。

### 4-3. zkEVMとは？

**EVM（Ethereum Virtual Machine）** とは、Ethereumのスマートコントラクトを実行する仮想マシンです。

**zkEVM**は、「**EVMの実行をZKで証明できる仮想マシン**」です。

#### 最もシンプルな実装アイデア（問題あり）

まず思いつく最もシンプルな方法：

```
「実行したい関数ごとにZK回路を作る」

add()関数用回路 ─┐
transfer()用回路 ─┤──▶ それぞれZKで証明
swap()用回路   ─┘
```

**問題点：**
- すでにデプロイされているコントラクトが使えず、すべて作り直しになる。
- Circom等の専用コードで一から書き直す必要がある。
- 1回ずつの実行結果をL1で検証するため、処理速度向上に繋がらない。

#### より良いアイデア（ベースアイデア）

```
Tx + コントラクトのEVMバイトコード
         │
         ▼
   ┌────────────┐
   │  ZK回路    │ ← EVMバイトコードを回路内で解釈・実行
   │            │
   └────────────┘
         │
         ▼
  更新後のステートルート + Proof
```

L1の検証者は：
1. バイトコードがL1にデプロイされているものと一致するか確認
2. Proofを検証

**問題点：**
- 回路の中でEVMバイトコードを解釈するため、非常に巨大な回路になる
- プルーフ生成の計算コストが膨大
- バイトコードをPublic Inputに含めるとL1検証時にデータサイズ問題

### 4-4. EVMバイトコードとOpcodeとは？

Solidityで書いたスマートコントラクトはコンパイルされてバイトコードになります。

```solidity
contract AAA {
    function add(uint a, uint b) public pure returns (uint256) {
        return a + b;
    }
}
```

これをコンパイルすると：
```
0x608060405234801561001057600080fd5b5060405160208061...
```

このバイトコードは**Opcode（オペコード）**の列です。EVMはOpcode一つ一つを順番に実行して状態を変化させます：

| Program Counter | Opcode | Stack | Memory | Storage |
|----------------|--------|-------|--------|---------|
| 53 | PUSH1 0x80 | [, , 80] | ... | ... |
| 54 | PUSH1 0x40 | [, 40, 80] | ... | ... |
| 55 | MSTORE | [, , ] | 0x40→0x80 | ... |
| 56 | CALLVALUE | [, , value] | ... | ... |

- **PUSH1**：スタックに値をプッシュ
- **MSTORE**：メモリに値を保存
- **CALLVALUE**：送金額をスタックにプッシュ

EVMはStack・Memory・Storageという3つの状態を持ちながら、Opcodeで状態遷移します。

### 4-5. zkEVMの回路内での処理

ZK回路内でやることは2つ：

#### ① EVM証明（Opcode処理の証明）
各Opcodeが正しく実行されたことを証明する。

例：`PUSH1 0x80` を実行したら、スタックの一番上に `0x80` が追加される。

#### ② 状態証明（State証明）
スタック・メモリの状態がOpcodeによって**正しく更新された**ことを証明する。

```
回路への入力：
  ┌────────────────────────────────────────┐
  │ EVMバイトコード（Opcode列）             │
  │ 状態遷移表（各Opcode実行後の状態）      │ ← 回路外で事前実行して用意
  │ Tx情報（関数への入力値）               │
  │ ステートツリー（変数の保存場所）        │
  └────────────────────────────────────────┘
              │
              ▼ ZK回路
  各行の状態遷移がOpcodeと整合しているか証明

  PC=53: PUSH1 0x80 → Stack = [..., 0x80] ✓  ← EVM証明
  PC=54: PUSH1 0x40 → Stack = [..., 0x40, 0x80] ✓  ← EVM証明
  PC=55: MSTORE → Memory[0x40] = 0x80, Stack = [...] ✓  ← EVM証明
```

> **重要：** 状態遷移表は回路外でEVMバイトコードを事前実行して取得し、Inputとして与えます。ZK回路は「その状態遷移が正しいか」を証明する役割です。

### 4-6. zkEVMの回路全体構成

実際のzkEVM（例：Scroll）の回路は非常に複雑で、以下のようなサブ回路で構成されます：

```
Transactions + Block Header
        │
        ▼
  ┌───────────────────────────────────────────────────────────┐
  │                        zkEVM 回路群                        │
  │                                                           │
  │  Public Input Circuit ──▶ Block Circuit                  │
  │                                │                         │
  │  Keccak Circuit ─────────────▶ Tx Circuit               │
  │  (ハッシュ計算)                  │                         │
  │                                ▼                         │
  │  Bytecode Circuit ──────▶ EVM Circuit ◀── State Circuit  │
  │  (バイトコード検証)         (Opcode実行)   (状態遷移検証)   │
  │                                │                         │
  │  MPT Circuit ─────────────────▶ Copy Circuit            │
  │  (Merkle Patricia Trie)        (データコピー検証)          │
  │                                                           │
  │  Precompile Circuits (前処理回路群)                        │
  └───────────────────────────────────────────────────────────┘
```

各回路が独立してProofを生成し、最終的に1つのProofにまとめられます。

### 4-7. プルーフ生成時のパフォーマンス向上技術

#### zk-unfriendlyな処理とは？

ZK回路内で特に計算コストが高い処理：

- **Keccak256ハッシュ**：EVMの標準ハッシュ関数だが、ビット演算が多くZK回路内では重い
- **Merkle Patricia Trie操作**：データの位置特定＋ハッシュ計算が必要でオーバーヘッド大

#### 対策①：再帰証明（Recursion）

```
Tx1のProof ─┐
Tx2のProof ─┤──▶ Aggregation Proof（集約証明）──▶ L1で検証
Tx3のProof ─┘
```

- 個々のProofを「Proof検証のZK回路」でまとめて1つのProofに集約する。
- 個別のProof生成は並列処理可能。

#### 対策②：フォールディング（Folding）

通常：各TxごとにProofを生成して検証。

フォールディング：複数のTxをまとめて**1つの制約（Constraint）**にしてから証明。

$$
\text{Constraint}(Tx_1) + \text{Constraint}(Tx_2) + \cdots \xrightarrow{\text{Folding}} \text{1つの制約} \xrightarrow{\text{証明}} \text{Proof}
$$

リカージョンよりもさらに計算コストの低減が期待できます。

#### 対策③：Lookupテーブル

計算を回路内でリアルタイムに行うのではなく、**事前計算した結果表（Lookup Table）**を参照する方法。

例：SHA256の計算が重い場合：
- 0 〜 $2^{32}-1$ の範囲の入力に対するSHA256結果をあらかじめ計算し、一覧表を作る。
- 回路内では計算ではなく「表を検索する」だけで済む。

$$
\text{通常}: x \xrightarrow{\text{SHA256計算（重）}} \text{SHA256}(x)
$$

$$
\text{Lookup}: x \xrightarrow{\text{表を参照（軽）}} \text{SHA256}(x)
$$

### 4-8. zkEVMの互換性レベル（Vitalikの分類）

すべてのEthereumの機能をそのままZK化するとプルーフ生成コストが非常に高くなります。各プロジェクトはトレードオフを選択しています：

| タイプ | 互換性 | 証明速度 | 説明 |
|--------|--------|----------|------|
| **Type 1** | Ethereum完全互換 | 非常に遅い | Ethereum自体を検証可能。L2だけでなくL1のスケーリングにも使える |
| **Type 2** | EVM互換（高） | 遅い | Ethereumのような機能性。ほぼ全アプリと互換 |
| **Type 2.5** | EVM互換（中） | 少し速い | ガスコストの変更のみ行う。少しの非互換性あり |
| **Type 3** | EVM互換（低） | 速い | Ethereumのような機能だが修正あり。ほぼ互換 |
| **Type 4** | スマートコントラクト言語互換 | 最も速い | Solidityを別のVMにコンパイル。多くのインフラ非共有 |

**互換性↑（証明が遅い） ←→ パフォーマンス↑（互換性低い）** のトレードオフがあります。

### 4-9. zkRollup/zkEVMまとめ

```
zkRollup/zkEVMの要点：
  ┌──────────────────────────────────────────┐
  │ Optimistic Rollupと同様にL2でTxをバッチ処理│
  │     ↓                                    │
  │ ZKにより「計算の正しさ」を数学的に保証      │
  │     ↓                                    │
  │ L1での検証コストを一定に抑える              │
  └──────────────────────────────────────────┘

課題：
  - Ethereumの機能をそのまま回路化するのは計算量的に困難
  - 互換性とパフォーマンスのトレードオフ
  - 各プロジェクト（Scroll, zkSync, Polygon zkEVM等）が異なるアプローチで解決中
```

## 5. 事例③：Email Wallet（ZKEmail）

### 5-1. 概要

**Email Wallet（ZKEmail）**は、**普段使っているメールアドレスでブロックチェーンウォレットを操作できる**ソリューションです。

メールを送るだけで：
- 暗号資産の送金
- スマートコントラクトの実行（Uniswapでのスワップなど）

ができます。

> 参考論文：Suegami, Sora, and Kyohei Shibano. "Contract Wallet Using Emails." 2023 IEEE International Conference on Blockchain and Cryptocurrency (ICBC). IEEE, 2023.

### 5-2. 基本的なアイデア：DKIMとは？

**DKIM（DomainKeys Identified Mail）**は、メールの改ざんを防止するための電子署名の仕組みです。

**DKIMの動作：**

```
送信者メールサーバー（例：example.jp）
  ① DNSにRSA公開鍵を事前登録
  ② 送信メールの本文とヘッダにRSA署名を付与
  ③ メールをSMTPで送信

受信者メールサーバー（例：example.com）
  ④ DKIM-SignatureヘッダのdパラメータでDNSに公開鍵を問い合わせ
  ⑤ 公開鍵でRSA署名を検証 → OK/NG
```

RSA署名の検証：

$$
\text{署名} = \text{RSA\_Sign}(\text{メール内容}, \text{秘密鍵})
$$

$$
\text{検証}: \text{RSA\_Verify}(\text{メール内容}, \text{署名}, \text{公開鍵}) = \text{true/false}
$$

**Gmail・Outlook・iCloudなど、現在普及しているほぼすべてのメールサービスがDKIMに対応**しています。

つまり、メールには「このメールがこの送信者から正しく送られた」という証明が付いています。

### 5-3. アイデアの核心

「メールにはDKIM署名（RSA）がついているなら、その署名をスマートコントラクトで検証して、資産の移動に使えるのでは？」

**問題：** RSA署名の検証をスマートコントラクト上でそのまま行うのは計算コストが非常に高い。

RSA署名の検証は以下の計算を含みます：

$$
m \equiv s^e \pmod{n}
$$

- $s$：署名
- $e$：公開指数
- $n$：RSAモジュラス（2048ビット等）
- $m$：メッセージのハッシュ値

この大きな数のべき乗計算（モジュラー冪乗）はガスを大量に消費します。

**ZKの解決策：** RSA署名の検証をZK回路内で行い、「正しく検証できた」というProofだけをスマートコントラクトに送る。

- ProofはどんなZK計算でも検証コストが一定 → L1での検証が安い！

### 5-4. システム構成

```
ユーザー
  │
  │ メールを送信（例："Transfer 0.1 ETH to yyy@domain.com"）
  ▼
送信メールサーバー（Gmail等）
  │
  │ DKIMによりRSA署名を付与
  ▼
リレイヤー（ZKEmailのサーバー）
  │
  │ 受信したメールを解析：
  │   - 送信者メールアドレスを抽出
  │   - メール本文から命令を解析（正規表現）
  │   - DKIM署名を用いてZKプルーフを生成
  │
  │ Txを作成：
  │   - 送信者（メールアドレスから決定されるウォレット）
  │   - 命令（送金先、金額等）
  │   - ZKプルーフ
  ▼
Ethereum スマートコントラクト
  │
  │ ZKプルーフを検証
  │ → OKなら命令を実行（送金、swap等）
  ▼
 完了
```

### 5-5. ZK回路で何を証明するか？

ZK回路が証明する内容：

1. **DKIM署名の正当性**：メールが本当にそのメールサーバーから送られたもの

$$
\text{RSA\_Verify}(\text{メール内容}, \text{DKIM署名}, \text{公開鍵}) = \text{true}
$$

2. **送信者アドレスの抽出**：メールヘッダから正しく送信者アドレスを取り出した

3. **命令の解析**：メール本文から正規表現で命令を正しく解析した

これらを**Private Inputでメールの生データを隠しながら**証明します。

**なぜ隠す必要があるのか？**
- メールには署名者の名前や本文など、プライバシーに関わる情報が含まれているため。
- ブロックチェーン（パブリック）に全文を公開したくない。

### 5-6. 送金例

```
Example 1：
メール本文：「Transfer 0.1 ETH to yyy@domain.com」
→ xxx@domain.com のウォレットから yyy@domain.com のウォレットへ 0.1 ETH 送金

Example 2：
メール本文：「Swap 1 ETH to DAI via Uniswap」
→ xxx@domain.com のウォレットで Uniswap を通じて ETH → DAI のスワップ実行
```

メッセージ形式は**正規表現**で柔軟に設定可能です。

### 5-7. メールサーバーへのトラストについて

この仕組みには前提条件があります：

> **メールサーバーがメールを改ざんしていないこと**を信頼する必要があります。

もしGmailがメールを改ざんしてDKIM署名を付け直したら、不正な命令が実行されてしまいます。これは「メールサーバーへのトラスト」と呼ばれます。

## 6. 事例④：Private IPFSのデータ存在証明

### 6-1. IPFSとは？

**IPFS（InterPlanetary File System）**は、ファイルを分散して保存するシステムです。

- ファイルごとに「**CID（Content Identifier）**」というユニークなIDが付与される。
- CIDはファイルの内容を元に計算されるハッシュ値。

$$
\text{CID} = \text{Hash}(\text{ファイル内容})
$$

**プライベートIPFS**は、特定のグループだけがアクセスできる閉じたIPFSネットワークです。

**用途例：** 医療情報・社内文書など機密情報を特定グループ内で分散管理。

### 6-2. 何が問題か？

プライベートIPFSで管理されたデータを**外部の人に配布したいとき**：

「本当にそのプライベートIPFSで管理されていたデータなのか？」を外部の人が確認できない。

さらに、データを加工（名前の埋め込み、特定情報の削除）してから配布したい場合：

「元のデータからこの加工データが作られた」ことを証明できない。

### 6-3. ZKによる解決

**ZK回路で以下を実現：**

```
[Private Input（秘密）]        [Public Input（公開）]
  画像暗号化データ        ──▶     IPFS CID
  画像暗号化鍵           ──▶     購入者名

          │
          ▼
    ZK回路（circom, Groth16）

  ┌───────────────────────────────────────────────┐
  │                                               │
  │  ① ハッシュ検証                               │
  │    Hash(画像暗号データ) == CID ?              │
  │    → 確かにIPFSに存在するデータであることを確認 │
  │                                               │
  │  ② 復号化                                    │
  │    共通鍵暗号で画像を復号                      │
  │                                               │
  │  ③ 名前の埋め込み                             │
  │    復号した画像に購入者名を文字として埋め込む   │
  │                                               │
  └───────────────────────────────────────────────┘

          │
          ▼
[Public Output（公開）]         [Proof（証明）]
  購入者名埋め込み済み画像    +   ZKプルーフ
```

**外部ユーザーが受け取るもの：**
1. 名前入りの加工済み画像
2. ZKプルーフ（この画像が正当なプロセスで生成された証拠）

**外部ユーザーが確認できること：**
- 画像が確かにプライベートIPFSに存在するCIDのデータから生成された
- 正しく名前が埋め込まれている
- 元の暗号化データや鍵を知らなくても検証可能

### 6-4. 計算コストの現実（当時の計測）

実際に論文で計測した結果：

| 画像サイズ | 非線形制約数 | ビルド時間 | Proof生成時間 |
|-----------|------------|-----------|-------------|
| 10×10 px | 35,407 | 128,979 ms（約2分） | 3,076 ms |
| 30×30 px | 263,450 | 470,872 ms（約8分） | 9,733 ms |
| 60×60 px | 1,013,483 | 1,257,418 ms（約21分） | 29,044 ms |
| 120×120 px | 4,036,913 | 6,754,928 ms（約112分） | 96,580 ms |

**実験環境：**
- Windows 11 / Ryzen 9 3950X / 128 GB RAM / Ubuntu 22.04 (WSL2)

**分かること：**
- 120×120ピクセルよりも大きい画像はRAM不足で処理不可
- ZK-Optimized実装でも120×120のビルドに約2時間弱かかる
- ZKの計算コストの重さ（zk-unfriendly処理の課題）がよく分かる事例

これが、ZKの大きな課題：「**プルーフ生成にかかる計算コストと時間**」を示しています。

## 7. まとめ

### ZKの2つの役割とその応用

| ZKの役割 | 応用事例 |
|----------|----------|
| **入力の秘匿化** | Zcash（送金者・金額を隠す）、Private IPFS（元データを隠して加工） |
| **検証コストの簡略化** | zkRollup/zkEVM（L2計算を低コストで検証）、ZKEmail（RSA署名検証を低コスト化） |

### ZKの限界と課題

| 課題 | 内容 |
|------|------|
| **プルーフ生成コスト** | オフチェーンでの証明生成に大きな計算資源が必要 |
| **秘匿の範囲** | 任意の情報を隠せるわけではなく、あくまで回路への入力値の一部のみ |
| **互換性とパフォーマンス** | zkEVMでは互換性を上げるほど証明が遅くなるトレードオフ |
| **信頼の前提** | ZKEmail ではメールサーバーへのトラストが必要 |

### 全体像の整理

```
ゼロ知識証明（ZK）
  │
  ├─▶ ブロックチェーンのインフラ改善
  │       ├─ Zcash：プライバシー保護通貨
  │       └─ zkRollup/zkEVM：スケーリング
  │
  └─▶ DApps（分散型アプリ）
          ├─ ZKEmail：メールでウォレット操作
          └─ Private IPFS：機密データの安全な配布
```

ZKは「ブロックチェーンの透明性という欠点」と「計算能力の限界」という2つの根本的な課題を補う技術として、今後のWeb3インフラに不可欠な存在になっています。
