## 1. 全体像：SNARKとは何か

### SNARKとは

**SNARK**（Succinct Non-interactive ARgument of Knowledge）とは、「ある計算を正しく行った」ということを、**計算そのものをすべて見せずに、短い証明だけで納得させる**仕組みです。

たとえば：
- 「私はこのパスワードを知っている」（パスワード自体は秘密のまま）
- 「この取引は正しく処理された」（取引の詳細を全部公開せずに）

を証明できます。

### SNARKを作るための2つの部品

```
┌────────────────────────────┐
│  多項式コミットメントスキーム │ ──┐
└────────────────────────────┘   │
                                 ├──→ [ ブレンダー ] ──→ 汎用回路向けSNARK
┌────────────────────────────┐   │
│  多項式インタラクティブ      │ ──┘
│  オラクル証明（IOP）        │
└────────────────────────────┘
```

この講義では主に**多項式コミットメントスキーム（特にFRI）**を詳しく説明します。

## 2. 多項式IOPと多項式コミットメントの仕組み

### 多項式IOP（Polynomial Interactive Oracle Proof）とは

**多項式IOP**は、証明者（Prover, P）と検証者（Verifier, V）が対話する仕組みで：

1. **P**がまず多項式 $h$ を「送る」（ただしVは $h$ の全体を受け取らない）
2. **V**は $h$ をある1点でだけ評価できる
3. その後、標準的な対話的証明を実行する

**重要な性質：**
- Pの計算量：回路サイズに対して**線形時間**
- 証明の長さ：**対数的（logarithmic）**

### 多項式コミットメントスキーム（Polynomial Commitment Scheme）とは

**コミットメント**とは「封筒に入れて渡す」ようなイメージです。

$$\text{P} \xrightarrow{\text{Com}(h)} \text{V}$$

具体的には：

| ステップ | 内容 |
|---------|------|
| ① コミット | P が多項式 $h$ に「コミット」し、短い文字列 $\text{Com}(h)$ をVに送る |
| ② チャレンジ | V が評価したい点 $x$ を選んで P に送る |
| ③ 証明 | P が $y = h(x)$ と、「$y$ が $\text{Com}(h)$ と整合している」という証明 $\pi$ を送る |
| ④ 検証 | V が $\text{Com}(h)$、$x$、$y$、$\pi$ を使って正当性を確認する |

**必要な性質：**
- PはVが信じられない評価値 $y$ に対して正当な証明 $\pi$ を作れない（**binding性**）
- $\text{Com}(h)$ と $\pi$ は短く、生成・検証が高速

## 3. SNARKの動物園：種類と分類

SNARKには多くの種類があり、多項式IOPと多項式コミットメントの組み合わせによって特性が変わります。

### 多項式IOPの3分類

| 種類 | 基盤 | 例 |
|-----|------|-----|
| クラス1 | インタラクティブ証明（IP） | — |
| クラス2 | マルチ証明者インタラクティブ証明（MIP） | — |
| クラス3 | 定数ラウンド多項式IOP | Marlin、PlonK |

一般に、上から順に P の計算コストが増え、証明サイズと V のコストが減る傾向があります。

### 多項式コミットメントの3分類

#### クラス1：ペアリング + 信頼できるセットアップ
- **例：** KZG10
- **特徴：** 定数サイズの評価証明（最小の証明サイズ）
- **欠点：** 透明でない（trusted setup必要）、耐量子コンピュータ性なし

#### クラス2：離散対数ベース
- **例：** IPA/Bulletproofs、Hyrax、Dory
- **特徴：** 透明（透過的 = trusted setup不要）
- **欠点：** 耐量子コンピュータ性なし
- **ホモモルフィック性あり**：複数のコミットメントを効率よく束ねられる

#### クラス3：IOP + ハッシュ
- **例：** FRI、Ligero、Brakedown、Orion
- **特徴：** 透明、かつ**耐量子コンピュータ（post-quantum）**
- **欠点：** 証明サイズが大きい（数百KB）

### 透明なSNARKのハイライト

| SNARK | 特徴 | 短所 |
|-------|------|------|
| Halo2（IPA使用） | 透明なSNARKの中で最短の証明 | Vが遅い |
| STARKs等（FRI使用） | 耐量子の中で最短の証明 | 証明が大きい（100s KB） |
| Spartan等（高速証明者用コミットメント） | 最速のP | 証明サイズが大きい |

### 非透明なSNARKのハイライト

| SNARK | 特徴 | 短所 |
|-------|------|------|
| Groth16（線形PCP） | 最短の証明（3群元）、最速V | 回路固有trusted setup、耐量子性なし |
| Marlin-KZG、PlonK-KZG | ユニバーサルtrusted setup | Groth16より証明が4〜6倍大きい |

## 4. FRI多項式コミットメントの詳細

FRI（**F**ast **R**eed-Solomon **I**nteractive Oracle Proof of Proximity）は、多項式コミットメントの主要な方式の一つです。

### 4.1 最初のアイデアとその問題点

#### 最初のアイデア

最も単純な方法：
- P が多項式 $q$ の**すべての点での評価値**をMerkleツリーにコミットする
- Vが評価点 $r$ を要求したら、P は対応する葉（leaf）と開封情報を送る

#### 2つの問題点

**問題1：計算量が体の大きさに比例してしまう**

体（field）$\mathbb{F}_p$ の大きさは $p \approx 2^{64}$ や $2^{128}$ になることがあります。
このため、$|\mathbb{F}|$ 個の評価値すべてを計算するのは現実的に不可能です。
→ **次数 $d$ に比例した計算量**が必要

**問題2：Vが $q$ の次数を検証できない**

$\text{Com}(h)$ を受け取っただけでは、Vは「本当に次数 $k-1$ 以下の多項式か」を確認できません。

### 4.2 1の冪根（Roots of Unity）

#### 1の冪根とは

$n$ 乗すると $1$ になる数のことを**$n$ 次の1の冪根**といいます：

$$x^n = 1 \iff x^n - 1 = 0$$

体 $\mathbb{F}_p$ 上での集合：

$$\Omega = \{1, \omega, \omega^2, \ldots, \omega^{n-1}\}$$

ここで $\omega$ は**原始 $n$ 次の1の冪根**（$\omega^n = 1$ となる最小の $n$ 次）。

#### 重要な性質

| 性質 | 内容 |
|-----|------|
| **乗法部分群** | $x, y \in \Omega$ ならば $xy \in \Omega$ |
| **$n$ が偶数の場合①** | $x \in \Omega$ ならば $x^2$ は $(n/2)$ 次の1の冪根 |
| **$n$ が偶数の場合②** | $x \in \Omega$ ならば $-x \in \Omega$ |
| **存在条件** | $|\Omega| = n$ $\iff$ $n$ が $p-1$ を割り切る |

#### なぜ1の冪根が重要なのか

写像 $x \mapsto x^2$ が $\Omega$ 上で**2対1（2-to-1）**になるからです。
これにより各ステップでドメインを半分に折りたたむことができます。

#### 有限体での例：$\mathbb{F}_{41}$（$p=41$）

$$\begin{aligned}
&1\text{ 次の1の冪根：} \{1\} \\
&2\text{ 次の1の冪根：} \{1, -1\} = \{1, 40\} \\
&4\text{ 次の1の冪根：} \{1, -1, 9, -9\} = \{1, 40, 9, 32\} \\
&8\text{ 次の1の冪根：} \{1, -1, 9, -9, 3, -3, 14, -14\}
\end{aligned}$$

#### なぜ $p = 2^{64} - 2^{32} + 1$ がよく使われるのか

この素数では $p - 1$ が $2^{32}$ で割り切れるため、最大 $n = 2^{32}$ まで1の冪根の集合が存在できます。FRI系のSNARKはこの体上で動作することが多いです。

#### 視覚化（複素数の場合）

複素数で考えると1の冪根は単位円上に等間隔に並びます。

```
    Im
 1.0│   ·  ·
    │ ·       ·
 0.5│·           ·
    │               
0.0 ┼─·─────────·──── Re
    │               
-0.5│·           ·
    │ ·       ·
-1.0│   ·  ·

  8次の1の冪根（例）
```

有限体では「円」の概念はありませんが、同じ代数的構造を持ちます。

### 4.3 FRIコミットメントの構造

**問題1の解決策：**

$q$ のすべての $p-1$ 個の評価値をコミットする代わりに、特定のサブセット $\Omega$ 上の評価値だけをコミットします。

$$|\Omega| = \rho^{-1} k$$

ここで：
- $k$ = 多項式の次数
- $\rho \leq 1/2$ = **レート**（Reed-Solomonコードのレート）
- $\rho^{-1} \geq 2$ = **FRIブローアップファクター**（blowup factor）

#### トレードオフ

$$\text{ブローアップファクター} \uparrow \implies \begin{cases} \text{P の計算コスト} \uparrow \\ \text{検証コスト} \downarrow \end{cases}$$

証明の長さ（ハッシュ値の数）：

$$\approx \frac{\lambda}{\log \rho^{-1}} \cdot \log_2^2(k)$$

ここで $\lambda$ はセキュリティパラメータ（「$\lambda$ ビットのセキュリティ」）。

#### 具体例：$\mathbb{F}_{41}$ 上の次数3多項式、$\rho^{-1} = 4$（$n=8$）

8次の1の冪根 $\Omega = \{1, -1, 9, -9, 3, -3, 14, -14\}$ 上の評価値をMerkleツリーにコミット：

```
  q(1)  q(-1)  q(9)  q(-9)  q(3)  q(-3)  q(14)  q(-14)   ← 葉（leaves）
   └──────┘    └──────┘     └──────┘     └──────────┘
  m₁=H(q(1),q(-1)) m₂=H(q(9),q(-9)) m₃=H(q(3),q(-3)) m₄=H(q(14),q(-14))
        └──────────────┘                └──────────────┘
       h₁ = H(m₁, m₂)                h₂ = H(m₃, m₄)
                    └────────────────────┘
                       k₁ = H(h₁, h₂)    ← Merkleルート（コミットメント）
```

Vは $k_1$ だけを受け取り、これが多項式 $q$ のコミットメントになります。

### 4.4 折りたたみフェーズ（Folding Phase）

**問題2の解決策：低次数テスト（Low-Degree Test）**

Vは「コミットしたベクトルが本当に次数 $k-1$ 以下の多項式の評価値か」を確認する必要があります。

FRIの低次数テストは：
1. **折りたたみフェーズ（Folding Phase）**：$\log k$ ラウンド
2. **クエリフェーズ（Query Phase）**：1ラウンド

に分かれています。

#### 折りたたみの数学的基礎

多項式 $q(X)$ を**偶数次・奇数次のパート**に分解します：

$$q(X) = q_e(X^2) + X \cdot q_o(X^2)$$

**例：** $q(X) = 1 + 2X + 3X^2 + 4X^3$ の場合

$$\begin{aligned}
q_e(X) &= 1 + 3X \quad \text{（偶数次の係数）}\\
q_o(X) &= 2 + 4X \quad \text{（奇数次の係数）}
\end{aligned}$$

確認：$q_e(X^2) + X \cdot q_o(X^2) = (1 + 3X^2) + X(2 + 4X^2) = 1 + 2X + 3X^2 + 4X^3$ ✓

VがランダムなフィールQ元素 $r$ を選んで送ると、**処方された折りたたみ（prescribed folding）**は：

$$q_{\text{fold}}(Z) = q_e(Z) + r \cdot q_o(Z)$$

**次数が半分になる：** $\deg(q_{\text{fold}}) = \lfloor (\deg(q) - 1) / 2 \rfloor$

#### 折りたたみの重要な公式

$x$ と $-x$ が $n$ 次の1の冪根で、$z = x^2$ のとき：

$$\boxed{q_{\text{fold}}(z) = \frac{r+x}{2x} \cdot q(x) + \frac{r-x}{-2x} \cdot q(-x)}$$

**証明の概略：**

$q(x) = q_e(z) + x \cdot q_o(z)$ と $q(-x) = q_e(z) - x \cdot q_o(z)$ から：

$$q_e(z) = \frac{q(x) + q(-x)}{2}, \quad q_o(z) = \frac{q(x) - q(-x)}{2x}$$

よって：

$$q_{\text{fold}}(z) = q_e(z) + r \cdot q_o(z) = \frac{q(x) + q(-x)}{2} + r \cdot \frac{q(x) - q(-x)}{2x}$$

整理すると上記の公式が得られます。

**直感的な理解：**
- $r = x$ のとき $q_{\text{fold}}(z) = q(x)$
- $r = -x$ のとき $q_{\text{fold}}(z) = q(-x)$
- つまり $q_{\text{fold}}(z)$ は $r$ の線形関数で、2点での値が決まれば一意に定まる

#### 折りたたみの繰り返し（$\mathbb{F}_{41}$ 上、次数3、$\rho^{-1}=4$）

```
【ラウンド1】
q(1)  q(-1)    q(9)  q(-9)    q(3)  q(-3)    q(14)  q(-14)
 │      │        │      │       │      │        │       │
 └──────┘        └──────┘       └──────┘        └───────┘
 ↓r₁の線形結合   ↓r₁の線形結合  ↓r₁の線形結合   ↓r₁の線形結合
B(1)            B(-1)          B(9)             B(-9)
  （4要素 → 8次の1の冪根の上の次数1多項式の評価値）

【ラウンド2】
B(1)   B(-1)        B(9)    B(-9)
  │      │            │       │
  └──────┘            └───────┘
  ↓r₂の線形結合        ↓r₂の線形結合
C(1)                 C(9)
  （2要素 → 4次の1の冪根の上の次数0多項式）

【最終】
1要素（定数） → 次数0多項式 = 定数関数
```

各ラウンドで：
- ドメインのサイズが**半分**に
- 多項式の次数も**半分**に
- P は各ラウンドの結果に対して**Merkleコミットメント**を送る

### 4.5 クエリフェーズ（Query Phase）

折りたたみフェーズが終わった後、VはPが正直に折りたたみを行ったかを確認します。

#### Pがズルをする方法

Pは次のようにズルをする可能性があります：
- ある折りたたみのステップで、「処方された線形結合」とは異なる値を送る
- これにより見かけ上の次数を人工的に下げることができる

#### クエリフェーズの手順

V は各折りたたまれたベクトルから約 $\frac{\lambda}{\log \rho^{-1}}$ 個のエントリをランダムに選び、それぞれが前のベクトルの対応する2エントリの**処方された線形結合**と一致しているか確認します。

**証明サイズ（V時間）：**

$$\approx \frac{\lambda}{\log \rho^{-1}} \cdot \log^2 k \quad \text{個のハッシュ値}$$

#### Ligero/Brakedown/OrionとFRIの比較

| 方式 | 考え方 | 証明サイズ | 特徴 |
|-----|-------|----------|------|
| Ligero等 | $d^{1/2}$ 個のベクトル（各長さ $d^{1/2}$）を1回折る | $\approx d^{1/2}$ | 線形時間Prover、フィールド非依存 |
| FRI | 長さ $O(d)$ の1ベクトルを対数回折る | $\frac{\lambda}{\log \rho^{-1}} \log^2 k$ | より小さい証明サイズ |

## 5. セキュリティ解析

### 設定

- P は多項式 $q$ の $\Omega$ 上の評価値のベクトル $w$ をMerkleコミット
- $\Omega$：$n$ 次の1の冪根（$n = \rho^{-1} k$）
- $q$ は次数 $k-1$ 以下と主張

### ハミング距離（Hamming Distance）

$$\delta = \text{（$q$ と最近傍の次数 $(k-1)$ 多項式 $h$ との「相対ハミング距離」）}$$

具体的には、$\delta$ は $\Omega$ の中で $q(x) \neq h(x)$ となる $x$ の割合。

### 主要定理

**主張：** P が $t$ 個の「FRI検証者クエリ」すべてに合格する確率は最大：

$$\frac{k}{p} + (1 - \delta)^t$$

- 第1項 $\frac{k}{p}$：「ラッキーな折りたたみ」を P が得る確率の上界
- 第2項 $(1-\delta)^t$：P が $t$ 回のクエリすべてで検出を逃れる確率

### 直感的な解釈

$\delta \approx 1 - \rho$（最良の場合）のとき、V が受け入れる確率：

$$\approx (1-(1-\rho))^t = \rho^t$$

つまり、**各クエリが約 $\log_2(1/\rho)$ ビットのセキュリティを提供**します。

**例：** $\rho = 1/4$ の場合
- 各クエリ：約2ビットのセキュリティ
- $\lambda = 128$ ビット必要なら：$t = 64$ クエリ必要
- コスト：各クエリ $\approx \log^2 k$ 個のハッシュ値

### 注意事項（Caveat）

上記の主張は $\delta \leq 1 - \rho^{1/2}$ まで**証明されています**が、$\delta \leq 1 - \rho$ まで成立することは**予想（conjecture）**です。現在のFRI実装のほとんどはこの予想に基づいてセキュリティを評価しています。

### セキュリティ解析の直感

**もし P が「ラッキーな折りたたみ」を得なかった場合：**

1. $q$ が次数 $(k-1)$ の多項式から $\delta$ だけ離れているなら、$q_e$ か $q_o$ の少なくとも一方は $(n/2)$ 次の1の冪根上で次数 $(k/2-1)$ の多項式から $\delta$ だけ離れている

2. ランダム線形結合 $q_{\text{fold}}(Z) = q_e(Z) + r \cdot q_o(Z)$ も、高い確率で $\delta$ だけ離れている

3. これが繰り返され、最終的な折りたたみ結果も次数0（定数）から $\delta$ だけ離れている

4. P は定数を最終的な折りたたみとして送らざるを得ないが、それは矛盾

5. したがって、どこかの折りたたみで P は不正直に折りたたんでいる

6. 各クエリは確率 $\delta$ でこの不正を検出する → $t$ 個全部検出を逃れる確率は $(1-\delta)^t$

## 6. FRIへの既知攻撃

### 攻撃の概要

以下の P の戦略は、**どんな $q$ に対しても**（次数 $k$ から最大限遠い場合でも）、確率 $\rho^t$ で FRI 検証者の全チェックを通過できます。

### 攻撃手順

1. P は $\Omega$ の中から $k = \rho n$ 個の要素からなる集合 $T$ を選ぶ
2. P はその $T$ 上で $q$ と一致する次数 $k-1$ の多項式 $s$ を計算する（これは常に可能）
3. 折りたたみフェーズでは $q$ ではなく $s$ を折りたたむ

**攻撃が成功する確率：**

V の $t$ 個のクエリが全て $T$ に含まれる確率 $= \rho^t$

### なぜこの攻撃が可能か

$|T| = k = \rho n = \rho |\Omega|$ なので、各クエリが $T$ に落ちる確率は $\rho$。
$t$ 個全部が $T$ に落ちる確率は $\rho^t$。

### この攻撃の意味

セキュリティパラメータ $\lambda$ のために、$\rho^t \leq 2^{-\lambda}$ が必要。
つまり $t \geq \lambda / \log_2(1/\rho)$ 個のクエリが必要。

---

## 7. FRIから多項式コミットメントへ

### FRIだけでは不十分な2つの問題

1. P は $\Omega$ 上の評価値にしかコミットしていない（フィールド全体ではない）
2. V は $q$ が「次数 $d$ から遠くない」ことしかわからない（完全に次数 $d$ 以下とは言えない）

### 解決策：KZGのアイデアを借用

**重要な事実（KZGコミットメントでも使われる）：**

次数 $d$ の多項式 $q$ について、「$q(r) = v$」は以下と同値：

$$q(X) - v = w(X)(X - r)$$

を満たす次数最大 $d-1$ の多項式 $w$ が存在する。

### FRI多項式コミットメントのプロトコル

V が $q(r) = v$ を確認したい場合：

V は関数

$$\frac{q(X) - v}{X - r}$$

に対して次数 $d-1$ の次数上界で **FRIの折りたたみ + クエリ手順を実行する**。

- V がクエリで $\Omega$ 上の点でこの関数の評価値が必要なとき → $q$ の同じ点でのクエリ1回で得られる

**セキュリティ保証：**

V のチェックを（非自明な確率で）通過するためには、$v$ は $q$ に最も近い次数 $d$ の多項式 $h$ の $r$ での評価値 $h(r)$ に等しくなければならない。

### 現実のFRIの使われ方

現在、FRIは「通常の多項式コミットメント」より弱い原始として使われていますが、SNARKのセキュリティには十分です：

- P は**1つの多項式に束縛される**のではなく、**少数の低次多項式の集合**に束縛される

---

## 8. Fiat-Shamir変換と具体的なセキュリティ

### Fiat-Shamir変換の復習

**目標：** 対話的（interactive）プロトコルを**非対話的（non-interactive）**に変換する

$$\text{公開コイン対話的プロトコル} \xrightarrow{\text{Fiat-Shamir}} \text{非対話的議論}$$

**仕組み：** V のランダムチャレンジ $\beta$ をハッシュ関数 $R$（**ランダムオラクル**）で置き換える：

$$\beta = R(x, \alpha)$$

ここで $x$ は公開入力、$\alpha$ は P の最初のメッセージ。

P は証明 $(\alpha, \beta, \gamma)$ を一度に送り、V が検証する。

### グラインディング攻撃（Grinding Attack）

**攻撃：** P は「ラッキーな $\beta$」が得られるまで $\alpha$ を繰り返し試す

```
P は試す：
  α₁ → β₁ = R(x, α₁) → ダメ
  α₂ → β₂ = R(x, α₂) → ダメ
  α₃ → β₃ = R(x, α₃) → ラッキー！ → この α₃ を使って証明を作る
```

### 対話的セキュリティ vs 非対話的セキュリティ

#### 対話的セキュリティ

$\lambda$ ビットのセキュリティで対話的にFRIを実行した場合：

**保証：** Pはハッシュ関数の衝突を見つけられない限り、V のチェックを確率 $2^{-\lambda}$ 以下でしか通過できない。

**重要な点：** P は V のチャレンジを知るためには実際に V と対話する必要がある！

**例：** $\lambda = 60$ ビットの場合
- P がV を騙すには、平均 $2^{30}$ 回以上拒否される必要がある
- Ethereum では1ブロックが12秒 → $2^{30}$ ブロック ≈ 400年
- 現実的に不可能

#### 非対話的セキュリティ（Fiat-Shamir適用後）

$\lambda$ ビットの対話的セキュリティのプロトコルにFiat-Shamirを適用した場合：

**保証：** $2^k$ 回ハッシュ評価を実行できる P は、確率 $2^{k-\lambda}$ で攻撃成功

- P はオフラインで「静かに」グラインディング攻撃を実行できる
- V と対話する必要がない

#### 現実のハッシュ計算コスト

| データポイント | 内容 |
|-------------|------|
| Bitcoin ネットワーク | 約1時間で $2^{80}$ 個のSHA-256ハッシュ |
| ブロック報酬 | 時間あたり約$1,000,000相当 |
| 2020年1月のSHA-1衝突 | $2^{64}$ 回のSHA-1評価 → GPU使用で$45,000 |
| $2^{70}$ 回のハッシュコスト | 約$3,000,000（2020年時点） |

→ **60ビットの対話的セキュリティは多くの文脈で十分だが、60ビットの非対話的セキュリティは高価値な文脈では不十分**

## 9. 多ラウンドプロトコルとFiat-Shamirの危険性

### 多ラウンドプロトコルへのFiat-Shamir適用の危険性

多ラウンドの対話的プロトコルにFiat-Shamirを適用すると、**セキュリティが壊滅的に失われる**可能性があります。

### 具体例：$\lambda$ 回繰り返す「空プロトコル」

**プロトコルの設定：**
- P が（無視される）ナンスを送る
- V がコインを投げ、表が出たらREJECT、裏が出たらACCEPT
- 1回の健全性誤差：$1/2$
- $\lambda$ 回繰り返すと健全性誤差：$(1/2)^\lambda = 2^{-\lambda}$

#### このプロトコルにFiat-Shamirを適用すると...

不正な P は $O(\lambda)$ 回のハッシュ評価で非対話的証明を偽造できます！

**攻撃手順（各ラウンドを個別に攻略）：**

```
ラウンド1:
  m₁ を試す → H(x, m₁) = 表 → ダメ
  m₁' を試す → H(x, m₁') = 裏（ラッキー！） → m₁' を固定
  平均2回の試行で成功

ラウンド2:
  m₂ を試す → H(x, m₁', m₂) = 表 → ダメ
  m₂' を試す → H(x, m₁', m₂') = 裏（ラッキー！） → m₂' を固定
  平均2回の試行で成功

...（同様に各ラウンドを個別に攻略）

合計: 平均 2λ 回（O(λ) 回）のハッシュ評価で全ラウンドを攻略！
```

**本来必要な試行回数（非対話的セキュリティが正しく機能していれば）：** $2^\lambda$ 回

**→ $2^\lambda$ → $O(\lambda)$ への壊滅的なセキュリティ低下！**

### ラウンドバイラウンド健全性（Round-by-Round Soundness）

この脆弱性を防ぐには、より強い健全性の概念が必要です。

**ラウンドバイラウンド健全性：**
- 各ラウンドで、攻撃者は「一度に大きな奇跡」が必要（各ラウンドで少しずつ幸運を積み重ねることができない）
- 形式的には：各ラウンドで「状態」が健全か不健全かが定義でき、不健全な状態から健全な状態への遷移確率が低い

| プロトコル | ラウンドバイラウンド健全性 |
|----------|------------------------|
| Sum-checkプロトコル（講義4） | ✅ 証明あり |
| Bulletproofs | ✅ 類似のものが証明あり |
| FRI | ❌ まだ証明されていない |

### FRIの現状

FRIは対数ラウンドの対話的プロトコルですが、現在は**常に非対話的（Fiat-Shamir適用後）**に展開されています。

ラウンドバイラウンド健全性はまだ証明されておらず、これは理論的な未解決問題ですが、実務的には一定の仮定のもとで使われています。

## 10. まとめ

### FRI多項式コミットメントの全体像

```
【コミットフェーズ】
q(X) （次数k-1以下）
  ↓ Ω（n=ρ⁻¹k 個の1の冪根）上で評価
(q(ω⁰), q(ω¹), ..., q(ωⁿ⁻¹))
  ↓ Merkleコミットメント
Com(q) = Merkleルートハッシュ

【低次数テスト（折りたたみ+クエリ）】
折りたたみフェーズ（log k ラウンド）:
  各ラウンドでVがランダムrを送り、Pがベクトルを半分に折りたたむ
  → 各ラウンドで次数・ドメインサイズが半減
クエリフェーズ:
  Vがλ/log(1/ρ) 個のランダム位置でチェック

【評価証明】
「q(r) = v」を証明するには:
  (q(X)-v)/(X-r) が次数d-1以下であることをFRIで確認
```

### セキュリティ・効率性のまとめ

| パラメータ | 値 |
|----------|-----|
| コミットメント時間 | $O(n \log n)$（FFTベース） |
| 証明サイズ | $\frac{\lambda}{\log \rho^{-1}} \cdot \log^2 k$ ハッシュ値 |
| 検証時間 | 証明サイズに比例 |
| 信頼できるセットアップ | 不要（透明） |
| 耐量子コンピュータ性 | ✅（ハッシュ関数のみに依存） |

### キーワード整理

| 用語 | 意味 |
|-----|------|
| **SNARK** | 簡潔な非対話的知識議論 |
| **多項式コミットメント** | 多項式に「封をして」後で評価を証明する仕組み |
| **FRI** | Fast Reed-Solomon IOP；ハッシュのみに基づく多項式コミットメント |
| **1の冪根** | $x^n = 1$ を満たす元；FRIの評価ドメイン |
| **折りたたみ** | 次数を半分にしながらドメインを縮小する操作 |
| **ブローアップファクター $\rho^{-1}$** | 評価ドメインの大きさと次数の比 |
| **Fiat-Shamir変換** | 対話的プロトコルを非対話的に変換する手法 |
| **グラインディング攻撃** | ラッキーなハッシュ値が出るまで試し続ける攻撃 |
| **ラウンドバイラウンド健全性** | 各ラウンドで独立したセキュリティが保たれる性質 |
| **Reed-Solomonコード** | 誤り訂正コード；FRIの理論的基盤 |
