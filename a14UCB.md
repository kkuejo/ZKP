## 1. 背景：マルチチェーンの世界

現在の暗号資産・Web3の世界は、**一つのブロックチェーンだけで完結していない**。

| チェーン | 主な用途 |
|----------|----------|
| Ethereum | スマートコントラクト、DeFi |
| BNB Smart Chain | 低コストのDeFi |
| Cosmos | アプリ固有チェーン群 |
| Solana | 高速決済 |
| Polygon | Ethereumのスケーリング |

それぞれのチェーンは「孤立した島」のようなもの。**異なるチェーン間でデータや資産を移動させる仕組み**が必要になり、これが「ブリッジ（Bridge）」である。

---

## 2. クロスチェーンブリッジとは何か

### 基本的なイメージ

```
送信チェーン C₁                         受信チェーン C₂
┌───────────────┐                      ┌───────────────┐
│  ブロック     │  ──GetState(Addr,Slot)→  │  スマート     │
│  チェーン     │  ←──────────────────    │  コントラクト │
└───────────────┘                      └───────────────┘
```

- **送信チェーン $\mathcal{C}_1$**: 元のデータや資産があるチェーン
- **受信チェーン $\mathcal{C}_2$**: データや資産を受け取るチェーン
- **GetState(Addr, Slot)**: アドレスとスロット番号を指定して、$\mathcal{C}_1$ の状態を $\mathcal{C}_2$ が参照する操作

### ブリッジに求められる性質

1. **汎用性（Generality）**: 多くのアプリケーションを支援できること
2. **効率性（Efficiency）**: 速くて安いこと
3. **信頼最小化（Trust Minimization）**: 特定の誰かを「信じなければならない」状況を極力排除すること ← **これが最重要**

## 3. 従来のブリッジの問題点：信頼できる中間者への依存

### 従来の3つのアプローチ

#### ① サイドチェーン方式（例：PolyNetwork、Axelar）

```
C₁ ──→ [サイドチェーン（独自バリデータ群）] ──→ C₂
```

- サイドチェーンのバリデータが $\mathcal{C}_1$ の状態を検証し、$\mathcal{C}_2$ に伝える
- **信頼前提**: バリデータの $\frac{2}{3}$ 以上が正直であること

#### ② 委員会方式（例：Wormhole、Ronin）

```
C₁ ──→ [マルチシグ委員会] ──→ C₂
```

- 指定された委員会メンバーが多数決で承認
- **信頼前提**: 委員会の $\frac{2}{3}$ 以上が正直であること

#### ③ オラクル方式（例：LayerZero）

```
C₁ ──→ [Oracle] + [Relayer（独立必要）] ──→ C₂
```

- **信頼前提**: OracleとRelayerが**互いに独立していること**（共謀しないこと）

### 共通の問題

$$\text{セキュリティ} = f(\text{中間者の誠実さ})$$

つまり、中間者が悪意を持つか、あるいは秘密鍵が漏洩するだけで**資産が盗まれる**。

| プロトコル | 被害時期 | 損失額 | 主な原因 |
|------------|----------|--------|----------|
| BSC Bridge | 2022年10月 | $568M | 秘密鍵漏洩 |
| Nomad | 2022年8月 | $200M | 秘密鍵漏洩 |
| Harmony | 2022年6月 | $100M | 秘密鍵漏洩 |

## 5. 根本解決のアプローチ：ライトクライアント検証

### ライトクライアント検証とは

ブロックチェーンのフルノードは全ての取引データを保持するが、**ライトクライアント**は「**ブロックヘッダーだけ**」を追跡することで、チェーンの正当性を検証できる。

BFT（Byzantine Fault Tolerant）系のコンセンサスを例にとると：

$$\text{ブロック } h_{t+1} \text{ が正当} \iff \text{バリデータ委員会の}\frac{2}{3}\text{以上が署名している}$$

### 既存のライトクライアント実装とその限界

#### Cosmos IBC

- 各チェーンのバリデータが相手チェーンのブロックヘッダーを検証
- **問題**: 各チェーンが IBC クライアントを実装しなければならない（拡張性の限界）

#### NEAR Rainbow Bridge

- Ethereumのスマートコントラクトとして $\mathcal{C}_1$ のライトクライアントを実装
- **問題**: オンチェーンでの検証コストが非常に高い

$$\text{Gas コスト} \propto \text{署名検証数} \times \text{1署名あたりのコスト}$$

これを解決するのが **zkBridge** である。

## 6. ゼロ知識証明（ZKP）の基礎

### 直感的説明

「**私はある秘密を知っている。でも秘密そのものは教えない。それでも、あなたに私が秘密を知っていることを証明できる。**」

### ZKPの3つの性質

| 性質 | 意味 |
|------|------|
| **完全性（Completeness）** | 正直な証明者は、正しい主張を必ず証明できる |
| **健全性（Soundness）** | 偽の主張について、不正な証明者が検証者を騙せる確率は無視できる |
| **ゼロ知識性（Zero-Knowledge）** | 検証者は「命題が真であること」以外の情報を一切得られない |

### SNARKs とは

**SNARK** = Succinct Non-interactive ARgument of Knowledge

$$\pi = \text{Prove}(C, x, w)$$

- $C$: 検証したい計算（回路）
- $x$: 公開入力
- $w$: 秘密の証人（witness）
- $\pi$: 証明（Proof）

$$\text{Verify}(C, x, \pi) \in \{\text{True}, \text{False}\}$$

**重要な特性**:
- 証明サイズが小さい（Succinct）
- 検証が高速（特に Groth16 は一定時間）
- インタラクションが不要（Non-interactive）

## 7. zkBridgeの仕組み

### 核心的アイデア

中間者（委員会など）を「**ゼロ知識証明**」に置き換える。

$$\text{「委員会メンバーが誠実」という仮定} \;\xrightarrow{\text{ZKPで置換}}\; \text{「数学的に検証可能な証明}\ \pi\text{」}$$

### システム構成

```
送信チェーン C₁          証明生成者（Prover）         受信チェーン C₂
┌──────────┐              ┌──────────────┐            ┌──────────────────┐
│          │  hₜ, hₜ₊₁  │              │  hₜ,hₜ₊₁,π │  Updater Contract│
│ ブロック  │ ──────────→ │   Prover     │ ──────────→│  （Verifier）    │
│ hₜ, hₜ₊₁│              │ （誰でも可）  │            └──────────────────┘
└──────────┘              └──────────────┘                     │
                         Block Header Relayer                    ↓
                                                    Application Contracts
```

### 証明 $\pi$ が示すもの

$$\pi \text{ は以下を証明する：} \quad \exists \sigma_1, \sigma_2, \ldots, \sigma_k \;\; \text{s.t.} \quad \text{Verify}_{pk_i}(\sigma_i, h_{t+1}) = 1 \;\; \forall i \in S$$

ここで：
- $\sigma_i$: バリデータ $i$ の署名
- $pk_i$: バリデータ $i$ の公開鍵
- $S$: 委員会の $\frac{2}{3}$ 以上を構成する部分集合
- $h_{t+1}$: 新しいブロックヘッダー

言い換えると：「**$\mathcal{C}_1$ 委員会の過半数以上が $h_{t+1}$ に署名している**」という事実を、署名を公開せずに証明する。

### Proverは誰でも良い

従来の方式と異なり、Proverは信頼される必要がない：
- 不正な証明を作ろうとしても、数学的に不可能
- 誰でも（Permissionless）参加できる
- 分散化・検閲耐性が担保される

## 8. 技術的課題と解決策：deVirgo と 2層証明合成

### 課題：SNARKは計算コストが高い

ブロックチェーンは「ZKフレンドリー」に設計されていない。

**具体的な数字**:

| 演算 | 算術回路のゲート数（概算） |
|------|--------------------------|
| EdDSA 署名検証 1回 | $\approx 2 \times 10^6$ ゲート |
| 1回の状態遷移（128署名） | $\approx 2.56 \times 10^8$ ゲート |

$$\text{単純な実装のコスト} \approx 128 \times 2{,}000{,}000 = 256{,}000{,}000 \text{ ゲート}$$

これを1台のマシンで証明しようとすると、現実的でない時間がかかる。

### 解決策：2層の証明合成（2-layer Proof Composition）

#### Layer 1：deVirgo（分散証明生成）

**Virgo**（IEEE S&P 2020で発表された証明システム）を分散化したもの。

```
大規模算術回路 C
       │
       ▼
┌────────────────────────────────────────┐
│  deVirgo Prover（分散マシン群）         │
│  Machine₁  Machine₂  ...  Machine_N    │
│  ┌──────┐ ┌──────┐       ┌──────┐    │
│  │ C/N  │ │ C/N  │  ...  │ C/N  │    │
│  └──────┘ └──────┘       └──────┘    │
└────────────────────────────────────────┘
       │  証明 π₁（比較的大きい）
       ▼
```

**データ並列性の活用**:
128署名の検証は互いに独立しているため、128台のマシンに分散できる：

$$T_{\text{deVirgo}}(N \text{ 台}) \approx T_{\text{sequential}} / N$$

- **100倍の高速化**（128台使用時）
- **通信コスト**：証明時間の20%以下に抑制

#### Layer 2：再帰検証（Recursive Verification）

deVirgoの証明 $\pi_1$ は大きい（約2MB）。オンチェーン検証には向かない。

**解決策**：「$\pi_1$ の検証プロセス」を新しい算術回路 $C_2$ として表現し、より効率的な **Groth16** で証明する。

```
π₁（deVirgo証明、大きい）
       │
       ▼
┌──────────────────────────────────────┐
│ deVirgo Verifier → 回路 C₂ に変換    │
│ （C₂は元の C よりずっと小さい）        │
└──────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────┐
│ Groth16 Prover                        │
│ 証明 π₂ を生成（定数サイズ：131 bytes）│
└──────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────┐
│ Groth16 Verifier（オンチェーンSC）    │
│ ガスコスト：221K（非常に低い）          │
└──────────────────────────────────────┘
```

| システム | 証明サイズ | 検証コスト | 証明速度 |
|---------|-----------|-----------|---------|
| deVirgo（単体） | $\approx$ 2MB | 高い（78M gas） | 速い |
| Groth16（単体） | 一定（131 bytes） | 低い（221K gas） | 遅い |
| **deVirgo + Groth16（2層）** | **131 bytes** | **221K gas** | **実用的** |

## 9. 性能評価：実際どのくらい速い？

### 評価結果（論文 Table 2 より）

| 署名数 | deVirgo時間(秒) | 再帰証明時間(秒) | 合計時間(秒) | 証明サイズ(再帰あり) | オンチェーン検証コスト |
|--------|----------------|----------------|------------|--------------------|--------------------|
| 8 | 12.52 | 4.90 | 17.42 | **131 bytes** | **221K gas** |
| 32 | 12.80 | 5.41 | 18.21 | **131 bytes** | **221K gas** |
| 128 | 13.28 | 5.49 | 18.77 | **131 bytes** | **221K gas** |

### 重要な観察

1. **証明時間がほぼ一定**：署名数が 8→128（16倍）に増えても、証明時間は 17.42→18.77秒（約8%増）にすぎない。これは分散化（deVirgo）の効果。

2. **証明サイズが一定**：再帰検証（Recursive Verification, RV）を使うと、署名数によらず証明は常に **131 bytes**。

3. **オンチェーン検証コストが低い**：**221K gas** は Ethereum での一般的なトランザクション（21K gas）の約10倍程度。ブリッジ検証としては非常に経済的。

$$\text{通信コスト（分散化）}: \frac{131.89 \text{ GB（合計）}}{128 \text{ 台}} = 1.03 \text{ GB/台} \quad \text{（管理可能な範囲）}$$

## 10. zkBridgeの拡張性：アプリケーション層

### 2層アーキテクチャ

```
┌──────────────────────────────────────────────────────────────┐
│  アプリケーション層（Application Layer）                        │
│  ユーザーが指定するクロスチェーンアプリケーション                  │
│  ┌─────────────────┐   ┌──────────────────┐                  │
│  │ Application     │   │ Application      │                  │
│  │ Contract (C₁側) │   │ Contract (C₂側)  │                  │
│  └─────────────────┘   └──────────────────┘                  │
│              ↑ Updater Contract が API を公開 ↑               │
├──────────────────────────────────────────────────────────────┤
│  ベース層（Base Layer）                                        │
│  ブロックヘッダー同期                                           │
│  C₁ ──→ [Block Header Relay Nodes] ──→ [Updater Contract]    │
│                                           （C₂にデプロイ）      │
└──────────────────────────────────────────────────────────────┘
```

### ベース層の役割

**Updater Contract**（C₂にデプロイ）:
- ブロックヘッダーリレーノードから $h_t, h_{t+1}, \pi$ を受け取る
- $\pi$ を検証し、検証済みの C₁ の最新状態を保管
- 上位アプリケーションに API を公開

### アプリケーション層で実現できること

1. **メッセージパッシング（Message Passing）**
2. **クロスチェーン資産転送/スワップ（Asset Transfer/Swap）**
3. **クロスチェーン NFT 相互運用（NFT Interoperability）**

## 11. ユースケース1：メッセージパッシング

### 概要

ユーザーが $\mathcal{C}_1$ に書いたメッセージ $m$ を、 $\mathcal{C}_2$ 上のアプリケーションが受け取る。

### システム構成

- $\mathcal{SC}_{\text{send}}$: $\mathcal{C}_1$ にデプロイされた送信用スマートコントラクト
- $\mathcal{SC}_{\text{verify}}$: $\mathcal{C}_2$ にデプロイされた検証用スマートコントラクト
- Updater Contract: $\mathcal{C}_2$ にデプロイ

### 処理フロー（6ステップ）

```
Step ①: アプリ → SC_send にメッセージ m を送信
Step ②: SC_send が C₁ にメッセージ m を記録
Step ③: Block Header Relay Nodes がヘッダーと証明をリレー [ベース層]
Step ④: Updater Contract が検証・受理 [ベース層]
Step ⑤: アプリが SC_verify を呼び出し（m と Merkle証明を提供）
Step ⑥: SC_verify が以下を実行:
         ① m を含むブロックを特定し、そのブロックがDAGのメインチェーン上にあることを確認
         ② m の Merkle証明を検証
         → 検証成功 → m が C₂ 上のアプリケーションで利用可能に
```

### Merkle証明とは

ブロックチェーンの状態はMerkleツリーで管理される。

$$\text{Merkle証明}: \pi_m = \{h_{\text{sibling}_1}, h_{\text{sibling}_2}, \ldots, h_{\text{root}}\}$$

「トランザクション $m$ がこのブロックに含まれている」ことを、ブロック全体を送らずに数学的に証明できる。

$$\text{Root} = \text{Hash}(\text{Hash}(m \| h_{\text{sib}_1}) \| h_{\text{sib}_2} \| \ldots)$$

検証者はルートハッシュと少数のハッシュ値だけで検証できる（O(log n) のデータ量）。

## 12. 多層防御（Defense-in-Depth）

### 発想

zkBridge はそれ単体でも強力だが、**複数の実装を組み合わせる**ことでさらに安全性を高められる。

### 具体的な方法

1. **Proof Diversity（証明多様性）**: 複数の ZKP システム（deVirgo, Groth16 以外も）を並列に使い、全てが一致した場合のみ承認
2. **N-version Programming**: 独立したチームが別々に実装した検証器を組み合わせる
3. **Optimistic 解との組み合わせ**: 通常時は ZKP で高速処理、問題発生時は楽観的ロールアップ的な異議申し立て期間を設ける

### Hashi（実装例）

$$\text{Hashi} = \text{EVM Header Oracle Aggregator}$$

`https://github.com/gnosis/hashi` で公開されている Gnosis チームの実装。複数のブリッジ実装からヘッダーを収集し、合意したもののみを採用する。

## 13. zkBridgeが可能にするその他の技術

### State Proof（状態証明）

$$\pi_{\text{state}} = \text{Prove}(\text{「ブロック集合 } B \text{ においてアドレス } a \text{ の状態が } s \text{ に変化した」})$$

- **Algorand State Proof** として実装例あり
- 暗号的に検証可能な状態変化の証明

### ZKベースのライトクライアント検証

- モバイルユースケースを含む効率的なライトクライアント検証
- **Celo Plumo**: zkSNARKを使った Celo ブロックチェーンのライトクライアント

### プライバシーチェーンへの拡張

zkBridge は基本的にはチェーン間の「正しさの証明」であるため、プライバシーチェーン（Tornado Cash的な匿名チェーン）との接続にも拡張可能。

## 14. まとめ

### zkBridge の位置づけ

| 比較項目 | 従来（委員会型） | zkBridge |
|---------|----------------|----------|
| セキュリティ根拠 | 委員会の誠実さへの信頼 | 数学的証明（暗号的健全性） |
| 証明サイズ | 署名数に比例（大きい） | **常に131 bytes（固定）** |
| オンチェーン検証コスト | 低い（マルチシグのみ） | **221K gas（十分低い）** |
| 証明生成コスト | なし | あり（分散化で対処） |
| 秘密鍵漏洩リスク | 高い | **なし（鍵が存在しない）** |
| 参加の開放性 | 許可制 | **誰でも参加可能** |

### zkBridgeの5つの強み
