## 1. 前提知識：SNARKとは何か

### SNARKのざっくりとした説明

**SNARK**（Succinct Non-interactive ARgument of Knowledge）とは、「ある計算を正しく実行した」ことを、計算結果を全部見せることなく、短い証明だけで証明できる暗号技術です。

**身近な例え：**
- 試験の採点係（検証者）に対して、「この答案を全部見なくても、合格点を取った」と証明できる仕組みのようなものです。
- ブロックチェーンでは、「このトランザクションは正しいルールに従って処理した」ことを、全データを開示せずに証明できます。

### SNARKを作る共通の設計図

この講義で繰り返し登場する重要な設計パターンがあります：

```
┌─────────────────────────┐     ┌─────────────────────────┐
│  多項式コミットメント       │     │  多項式対話的オラクル証明  │
│  (Polynomial Commitment) │ ──▶ │  (Polynomial IOP)       │
│  Scheme                 │  ＋ │                         │
└─────────────────────────┘     └─────────────────────────┘
                   ▼（ブレンダーで混ぜる）
        ┌─────────────────────────┐
        │  一般回路向けSNARK       │
        │  (SNARK for general     │
        │   circuits)             │
        └─────────────────────────┘
```

つまり、**多項式コミットメント**と**多項式IOP**の2つの部品を組み合わせることで、汎用的なSNARKが作れます。

## 2. 多項式コミットメントとは何か

### コミットメントの直感的な説明

**コミットメント**とは、簡単に言えば「封筒に入れて封をする」操作です。

- **コミット（Commit）**：答えを封筒に入れて封をする（中身は見せない）
- **オープン（Open）**：必要なときだけ封を開けて中身を見せる

封をした後は中身を変えられない（**束縛性 / Binding**）が、封をしている間は中身が分からない（**秘匿性 / Hiding**）というのが重要な性質です。

### 多項式コミットメントの仕組み

多項式 $f(x)$ に対するコミットメントでは：

1. **証明者（Prover）**が多項式 $f(x)$ を持っている
2. 証明者は $f(x)$ の「指紋」となるコミットメント $\text{com}_f$ を検証者に送る
3. **検証者（Verifier）**がある点 $u$ を指定する
4. 証明者は $v = f(u)$ という評価値と、それが正しいことを示す証明 $\pi$ を返す
5. 検証者は $\text{com}_f$、$u$、$v$、$\pi$ だけを使って正しさを確認する

```
keygen(λ, F) → gp  （大域パラメータを生成）

証明者                              検証者
 f(x) ∈ F  ──commit(f)→ com_f──▶  com_f
            ←─────── u ──────────
eval(gp, f, u) → v, π  ──────▶    verify
```

**なぜ有用か？**
検証者は多項式 $f(x)$ 全体を見ることなく、特定の点での評価値が正しいかどうかを確認できます。

## 3. 前回の復習：KZG多項式コミットメント

### KZGスキームの概要

KZG（Kate-Zaverucha-Goldberg）は、**楕円曲線ペアリング**という数学的仕組みに基づいた多項式コミットメントです。

**設定：**
- 次数 $d$ 以下の単変数多項式を扱う
- グローバルパラメータ：$gp = (g, g^\tau, g^{\tau^2}, \ldots, g^{\tau^d})$
  - ここで $g$ は楕円曲線の生成元、$\tau$ は秘密の値（**信頼されたセットアップ**が必要）

**コミットの仕組み：**

多項式 $f(x) = a_0 + a_1 x + a_2 x^2 + \cdots + a_d x^d$ に対して：

$$\text{com}_f = g^{f(\tau)} = g^{a_0} \cdot (g^\tau)^{a_1} \cdot (g^{\tau^2})^{a_2} \cdots$$

これは $\tau$ の値を直接知らなくても、$g^{\tau^i}$ があれば計算できます。

**評価の証明：**

点 $u$ での評価 $f(u) = v$ を証明するため、商多項式：

$$q(x) = \frac{f(x) - f(u)}{x - u}$$

を使います。これは $f(u) = v$ であれば必ず割り切れます。

証明は $\pi = g^{q(\tau)}$ で、検証は次のペアリング等式で行います：

$$e(\text{com}_f / g^v,\ g) = e(g^{\tau - u},\ \pi)$$

### 離散対数に基づくスキームの比較

| スキーム | 証明者計算量 | 証明サイズ | 検証者計算量 | 信頼されたセットアップ | 暗号基盤 |
|---------|------------|----------|------------|-------------------|--------|
| Bulletproofs | $O(\lambda \cdot d)$ | $O(\lambda \log d)$ | $O(\lambda \cdot d)$ | 不要 | 離散対数 |
| Hyrax | $O(\lambda \cdot d)$ | $O(\lambda \sqrt{d})$ | $O(\lambda \sqrt{d})$ | 不要 | 離散対数 |
| Dory | $O(\lambda \cdot d)$ | $O(\lambda \log d)$ | $O(\lambda \log d)$ | 不要 | ペアリング |
| Dark | $O(\lambda \cdot d)$ | $O(\lambda \log d)$ | $O(\lambda \log d)$ | 不要 | 未知の位数群 |

## 4. 誤り訂正符号の背景知識

### なぜ誤り訂正符号か？

この講義では、楕円曲線などの代わりに**誤り訂正符号**を使った全く異なるアプローチを紹介します。

**動機（メリット）：**
- ✅ **耐量子計算機安全性**：量子コンピュータにも（おそらく）安全
- ✅ **群指数演算が不要**：証明者はハッシュ、加算、乗算だけを使う（高速！）
- ✅ **小さいグローバルパラメータ**：信頼されたセットアップが不要

**デメリット：**
- ❌ **証明サイズが大きい**：数十MBになることもある
- ❌ **準同型性がない**：証明の集約が難しい

### 誤り訂正符号とは？

**誤り訂正符号**とは、データを送信する際に誤りが生じても、元のデータを復元できるようにする仕組みです。

**直感的な例え：**
- CDが傷ついても音楽が再生できるのは誤り訂正符号のおかげ
- QRコードが一部汚れていても読み取れるのも同じ原理

**形式的な定義：$[n, k, \Delta]$ 符号**

$$\text{Enc}(m): \mathbb{F}^k \to \mathbb{F}^n$$

- $m$：長さ $k$ のメッセージ
- $\text{Enc}(m)$：長さ $n$ の符号語（codeword）（$n > k$）
- $\Delta$：**最小ハミング距離**（任意の2つの符号語が異なる位置の数の最小値）

```
メッセージ (長さk)  ──Enc──▶  符号語 (長さn)
  000...0 (k個)               000...000...0 (n個)
  100...0 (k個)               110...010...1 (n個)
                              ↑この2つは最低Δ箇所で異なる
```

### 繰り返し符号の例

$k = 2$（2ビット）、$n = 6$（6ビット）の繰り返し符号：

| メッセージ | 符号語 |
|----------|--------|
| $00$ | $000000$ |
| $01$ | $000111$ |
| $10$ | $111000$ |
| $11$ | $111111$ |

**最小距離** $\Delta = 3$（任意の2符号語は少なくとも3ビット異なる）

→ 1ビットの誤りがあっても訂正可能！

例：$010111 \to 01$（$01$に最も近い符号語 $000111$ にデコード）

### レート（符号化率）と相対距離

$$\text{レート} = \frac{k}{n}, \quad \text{相対距離} = \frac{\Delta}{n}$$

**トレードオフ**：レートを高くすると（情報を詰め込むと）、距離が小さくなり（誤り訂正能力が下がる）。

例：繰り返し符号は $\text{レート} = \frac{1}{a}$、相対距離 $= \frac{1}{k}$

### 線形符号

**線形符号**は、任意の2つの符号語の線形結合もまた符号語になる、という性質を持ちます。

$$\text{Enc}(m) = m \cdot G$$

ここで $G$ は**生成行列（generator matrix）**です。

**重要な性質：**
- 最小距離 = 最小重み（ゼロでない位置の数が最少の符号語の重み）

### Reed-Solomon符号

**Reed-Solomon符号**は実際に広く使われている重要な符号です（CDの誤り訂正にも使われています）。

**符号化の手順：**

1. メッセージ $(m_0, m_1, \ldots, m_{k-1}) \in \mathbb{F}_p^k$ を、次数 $k-1$ の多項式として解釈：

$$f(x) = m_0 + m_1 x + m_2 x^2 + \cdots + m_{k-1} x^{k-1}$$

2. $n$ 個の点 $(\omega, \omega^2, \ldots, \omega^n)$ での評価値を符号語とする：

$$\text{Enc}(m) = (f(\omega), f(\omega^2), \ldots, f(\omega^n))$$

ここで $\omega$ は $n$ 乗根（$\omega^n = 1 \mod p$）

**最小距離：**
$$\Delta = n - k + 1$$

なぜなら、次数 $k-1$ の多項式は高々 $k-1$ 個の根しか持てないから、2つの異なる多項式は高々 $k-1$ 個の点で一致します。

例：$n = 2k$ のとき、レート = $\frac{1}{2}$、相対距離 = $\frac{1}{2}$

**符号化時間：** 高速フーリエ変換（FFT）を使って $O(n \log n)$

## 5. 誤り訂正符号に基づく多項式コミットメント

### ステップ0：多項式係数を行列に配置する

次数 $d$ の多項式 $f(x)$ の係数を $\sqrt{d} \times \sqrt{d}$ の行列に並べます：

$$f(x) = \sum_{i=1}^{\sqrt{d}} \sum_{j=1}^{\sqrt{d}} f_{i,j} \cdot x^{i-1+(j-1)\sqrt{d}}$$

行列の形：

$$\begin{pmatrix} f_{1,1} & f_{1,2} & \cdots & f_{1,\sqrt{d}} \\ f_{2,1} & f_{2,2} & \cdots & f_{2,\sqrt{d}} \\ \vdots & & \ddots & \vdots \\ f_{\sqrt{d},1} & f_{\sqrt{d},2} & \cdots & f_{\sqrt{d},\sqrt{d}} \end{pmatrix}$$

### ステップ0.5：多項式評価を行列・ベクトル積に還元する

点 $u$ での評価は次のように分解できます：

$$f(u) = \underbrace{[1, u, u^2, \ldots, u^{\sqrt{d}-1}]}_{\vec{u}_L} \times \underbrace{F}_{\text{係数行列}} \times \underbrace{\begin{pmatrix} 1 \\ u^{\sqrt{d}} \\ u^{2\sqrt{d}} \\ \vdots \\ u^{(\sqrt{d}-1)\sqrt{d}} \end{pmatrix}}_{\vec{u}_R}$$

これは「ベクトル-行列積」の問題になります！

### ステップ1：行列を符号化してコミットする

**各行を線形符号で符号化します：**

```
元の行列 (√d × √d):          符号化後の行列 (√d × n):
┌────────────────┐            ┌──────────────────────────┐
│ f_{1,1} ...   │  ──Enc──▶  │ ●●●●●●●●●●●●●●●●●●●●●● │
│ f_{2,1} ...   │            │ ●●●●●●●●●●●●●●●●●●●●●● │
│    ⋮          │            │            ⋮             │
└────────────────┘            └──────────────────────────┘
  k = √d 列                     n 列（n > √d）
```

**各列のMerkleツリーによるコミットメント：**

Merkleツリーとは、多数のデータをまとめてハッシュ化して1つの「指紋」（ルートハッシュ）にまとめる仕組みです。

```
列1   列2   列3   列4
 ●     ●     ●     ●
  \   /       \   /
  m1=H(●,●)  m2=H(●,●)
      \           /
     k1=H(m1, m2)   ← これが1つのコミットメント H
```

列 $j$ を開示するには、その列の値と「認証パス（Merkleパス）」を送れば、$H$ から正しさが確認できます。

**全列の指紋を1つのルートハッシュ $H$ にまとめてコミットとします。**

### ステップ2：近接性テスト（Proximity Test）

**目的：** コミットした行列が本当に各行がcodewordになっているかを確認する

**プロトコル：**

1. 検証者がランダムな係数 $r_1, r_2, \ldots, r_{\sqrt{d}}$ を送る
2. 証明者は全行の線形結合を計算：

$$w = [r_1, r_2, \ldots, r_{\sqrt{d}}] \times F$$

3. 証明者は $w$ を検証者に送る
4. 検証者はいくつかのランダムな列 $j$ を指定し、次の3点を確認：
   - **Check 1：** $w$ は正しい符号語（codeword）か？
   - **Check 2：** 開示された列はMerkleツリーのコミットメント通りか？
   - **Check 3：** 各列 $j$ と $r$ の内積が $w_j$ と一致するか？

**なぜこれが機能するか（健全性の直感）：**

証明者が不正をしようとした場合：
- もし行列の線形結合 $w$ が正しく計算されていれば、$w$ はcodewordでないはず → Check 1で検出
- もし $w$ が偽りであれば、多くの列で正しい値と異なる → Check 2とCheck 3で確率的に検出

**正式な健全性（Ligero [AHIV'2017]）：**

行列 $C$ が任意のcodewordから $e$ 離れている（$e < \frac{\Delta}{4}$）とき：

$$\Pr[w = r^T C \text{ が } e\text{-近い}] \leq \frac{e+1}{|\mathbb{F}|}$$

$w$ がcodewordから $e$ 離れているとき、$t$ 個のランダムな列でCheck 3をパスする確率：

$$\Pr[\text{Check 3 が全て成立}] \leq \left(1 - \frac{e}{n}\right)^t$$

### ステップ3：一貫性テスト（Consistency Test）

**目的：** $f(u)$ の評価値が正しいことを確認する

$u$ での評価を思い出すと：

$$f(u) = \underbrace{[1, u, u^2, \ldots, u^{\sqrt{d}-1}]}_{\vec{u}_L} \times F \times \vec{u}_R$$

**プロトコル：**

1. 証明者は $m = \vec{u}_L \times F$ を計算して送る
2. 検証者はいくつかのランダムな列 $j$ を指定し：
   - **Check 3'：** $\vec{u}_L$ と各列の内積が $m_j$ と一致するか？
3. 検証者は自分で $m$ を符号化し、開示された列の値と照合する
4. 最終的に $f(u) = m \cdot \vec{u}_R$ を計算して確認

**ポイント：** 近接性テストにより、行列は正しい符号語に「近い」ことが保証されているので、抽出器（extractor）を使って本物の多項式 $F$ を復元でき、$\vec{u}_L \times F = m$ が高確率で成立することが証明できます。

### まとめ：線形符号に基づく多項式コミットメント

**プロトコル全体：**

1. **Keygen**：ハッシュ関数をサンプリング（透明なセットアップ！）
2. **Commit**：係数行列の各行を線形符号で符号化、Merkleツリーでコミット
3. **Eval & Verify**：
   - 近接性テスト：全行のランダム線形結合 → $t$ 個のランダム列でチェック
   - 一貫性テスト：$\vec{u}_L \times F = m$ をエンコードして $t$ 個の列でチェック
   - $f(u) = m \cdot \vec{u}_R$

### 性能特性

| 項目 | Reed-Solomon符号使用時 | 線形時間符号使用時 |
|-----|---------------------|-----------------|
| Keygen | $O(1)$、透明！ | $O(1)$、透明！ |
| コミット（符号化） | $O(d \log d)$ | $O(d)$ |
| コミット（Merkle） | $O(d)$ ハッシュ | $O(d)$ ハッシュ |
| コミットサイズ | $O(1)$ | $O(1)$ |
| Eval | $O(d)$ | $O(d)$ |
| 証明サイズ | $O(\sqrt{d})$ | $O(\sqrt{d})$ |
| 検証者時間 | $O(\sqrt{d})$ | $O(\sqrt{d})$ |

**実測値（[GLSTW'21]、次数 $d = 2^{25}$、線形時間符号使用時）：**

| 項目 | 値 |
|-----|---|
| コミット | 36秒 |
| Eval | 3.2秒 |
| 証明サイズ | 49 MB |
| 検証者時間 | 0.7秒 |

証明サイズが大きいのが現在の課題です。

## 6. 線形時間符号化可能なコード

### なぜ線形時間符号化が重要か？

Reed-Solomon符号の符号化は $O(n \log n)$ かかります。これをさらに速くして $O(n)$（線形時間）にできれば、証明者の計算時間を大幅に削減できます。

### Spielman符号の基本アイデア

**Spielman [1996]** と **Druk-Ishai [2014]** が提案した線形時間符号化可能なコードは、**展開グラフ（Expander Graph）** を利用します。

### 損失なし展開グラフ（Lossless Expander）

**展開グラフとは：**

2部グラフ（左ノード集合 $L$、右ノード集合 $R$ があるグラフ）で、「情報が効率よく広がる」性質を持つものです。

**パラメータ：**
- 左ノード数：$|L| = k$
- 右ノード数：$|R| = \alpha k$（$\alpha$ は定数）
- 各左ノードの次数：$g$（各左ノードはちょうど $g$ 本のエッジを持つ）

**損失なし展開性（Lossless Expansion）：**

左ノードの任意の部分集合 $S$（$|S| \leq \frac{\delta |L|}{g}$）に対して：

$$|\Gamma(S)| \geq (1 - \beta) g |S|$$

ここで $\Gamma(S)$ は $S$ の近傍（neighbor）の集合、$\beta \to 0$, $\delta \to \alpha$

**直感：** 小さい集合 $S$ のノードが持つエッジは「ほぼ全て」異なる右ノードに繋がっている（衝突が少ない）。

```
左ノード（メッセージ）       右ノード（圧縮情報）

   ●  ───────────────▶  ●
   ●  ─────────────▶  ●
   ●  ──────────▶  ●
   ●  ──────────────────▶  ●
   ...                    ...
  k個                   αk個
  各左ノードからg本のエッジ
```

### 再帰的符号化アルゴリズム

**全体構造：**

```
メッセージ m (長さk)
    │
    ├──────────────────▶  m（そのままコピー）
    │
    ▼ 展開グラフ1（α=1/2）
   m1（長さk/2）
    │
    ▼ 符号化（長さk/2 → 長さ2k）
   c1（長さ2k）
    │
    ├──────────────────▶  c1
    │
    ▼ 展開グラフ2（α=1/2）
   c2（長さk）
    │
    ▼
最終符号語 c = m ∥ c1 ∥ c2 （長さ 4k）
```

**具体的な手順（$k$ → $4k$ への符号化、レート $\frac{1}{4}$）：**

1. メッセージ $m$（長さ $k$）を展開グラフ（$\alpha = \frac{1}{2}$）に通して $m_1$（長さ $\frac{k}{2}$）を得る
2. $m_1$ を再帰的に符号化して $c_1$（長さ $2k$）を得る
3. $c_1$ を展開グラフに通して $c_2$（長さ $k$）を得る
4. 符号語 $c = m \| c_1 \| c_2$

この処理を再帰的に $k/2, k/4, \ldots$ と繰り返し、定数サイズになったらReed-Solomon符号などで処理します。

**計算量の分析：**

各レベルの計算量は $O(k)$（グラフの辺を辿るだけ）で、レベル数は $O(\log k)$ ですが、各レベルのサイズが半減するため合計は：

$$T(k) = O(k) + T(k/2) = O(k)$$

つまり**線形時間** $O(k)$ で符号化できます！

### 符号の距離の証明

この再帰的符号は定数相対距離を持ちます：

$$\Delta' = \min\left\{\Delta, \frac{\delta}{4g}\right\}$$

**証明の概略（Druk-Ishai [2014]）：**

符号語 $c = m \| c_1 \| c_2$ の最小距離を示すため、$c$ のハミング重みが $4k\Delta'$ 以上であることを示します：

**ケース1：** $m$ の重み（非ゼロ要素数）が $4k\Delta'$ より大きい → 符号語の重みは $4k\Delta'$ 以上 ✓

**ケース2：** $m$ の重み ≤ $4k\Delta'$ の場合：
- 非ゼロの $m$ のノード集合を $S$ とする（展開グラフの条件を満たすくらい小さい）
- 損失なし展開性より、$S$ の近傍 $\Gamma(S)$ の少なくとも1つは $S$ 内に唯一の近傍を持つ（= 「ユニーク」な近傍が存在）
- したがって $m_1$ は非ゼロ → $c_1$ の重みは $2k\Delta$ 以上（再帰的仮定より）

**ケース2a：** $c_1$ の重み > $4k\Delta'$ → ✓

**ケース2b：** $c_1$ の重みが小さい場合：
- 再び展開グラフを使って $c_2$ の重みが $2k\Delta'$ 以上であることを示す → ✓

### 展開グラフの構成方法

- **明示的構成：** [Capalbo-Reingold-Vadhan-Wigderson 2002] が明示的な損失なし展開グラフを構成（ただし隠れた定数が大きい）
- **ランダムサンプリング：** $1/\text{poly}(n)$ の失敗確率でランダムに生成

### 改良版の線形時間符号

| 論文 | 改良点 |
|-----|-------|
| Brakedown [GLSTW'21] | ランダム総和を使い、より良い具体的距離解析 |
| Orion [Xie-Zhang-Song'22] | グラフの最大密度による展開グラフテストで失敗確率を negligible（無視できるほど小）に |

### 全体まとめ

**誤り訂正符号に基づく多項式コミットメントの利点：**

| 性質 | 詳細 |
|-----|-----|
| ✅ 透明なセットアップ | $O(1)$ のグローバルパラメータ、信頼機関不要 |
| ✅ 線形時間証明者 | $O(d)$ の体加算・乗算のみ |
| ✅ 耐量子計算機安全性（見込み） | 楕円曲線に依存しない |
| ✅ 体非依存 | どんな有限体でも使える |
| ❌ 証明サイズ | $O(\sqrt{d})$、実測で数十MB（課題） |

**次回（FRIとSTARK）へのつながり：**
FRI（Fast Reed-Solomon Interactive Oracle Proof）は同様のアイデアをさらに発展させたもので、STARKシステムの基礎となっています。

## 付録：用語集

| 用語 | 説明 |
|-----|-----|
| **SNARK** | 簡潔な非対話型知識論証（Succinct Non-interactive ARgument of Knowledge） |
| **多項式コミットメント** | 多項式を「封印」して、後から特定の点の評価値を証明できる仕組み |
| **KZG** | Kate-Zaverucha-Goldbergの多項式コミットメント（楕円曲線ベース） |
| **誤り訂正符号** | データを冗長化して誤りに強くする仕組み |
| **Reed-Solomon符号** | 多項式評価に基づく誤り訂正符号 |
| **ハミング距離** | 2つの符号語が異なる位置の数 |
| **Merkleツリー** | ハッシュ関数を使って大量データを1つの指紋にまとめる仕組み |
| **展開グラフ** | 情報が効率よく「広がる」性質を持つグラフ |
| **近接性テスト** | コミットしたデータが本当に有効な符号語に近いか確認するテスト |
| **一貫性テスト** | 評価値が行列の内積と整合しているか確認するテスト |
| **IOP** | 対話的オラクル証明（Interactive Oracle Proof） |
| **テンソル積** | $u \otimes u' = u_L \otimes u_R$（多次元への自然な拡張） |
